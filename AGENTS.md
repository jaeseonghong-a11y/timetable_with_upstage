## 1. 프로젝트 한 줄 정의

성균관대 학부생 대상 **시간표 조합 추천 서비스**. Upstage AI Document Builders Challenge 출품작 (데모데이 2026-07-25).
**핵심 차별점:** 개설강좌를 긁는 데서 끝나지 않고, **강의계획서 PDF를 Upstage Document Parse/Extract로 구조화**해서
"이 시간표 조합은 과제가 몰려있다 / 시험이 겹친다 / 평가방식이 이렇다"까지 예측해준다.

**절대 잊지 말 것:** 이 대회의 필수 조건은 **Upstage 제품(Parse/Extract/Solar)이 서비스의 심장**이어야 한다는 것.
크롤링·조합계산은 조연이고, Upstage 문서파싱이 주연이다. 이 균형이 무너지면 대회 부적합이다.

### 데모데이 평가 기준 의사결정 게이트 (필수)

공식 100점 루브릭과 공지 원문은 `docs/08_데모데이_평가항목_루브릭.md`를 단일 기준으로 삼는다.
특히 정보 처리 파이프라인(20점), 정보 처리의 깊이·정교성(15점), Upstage 적합성·기여도(20점)를
기능 개수보다 우선해 증명한다.

큰 기능·데이터 구조·우선순위를 결정하기 전에 다음을 확인한다.

1. 어떤 정보를 어디서, 왜 가져와 어떤 결과로 연결하는가?
2. 입력 → 변환 → 검증 → 사용자 확인 → 결과를 설명할 수 있는가?
3. 범용 LLM만으로 대체하기 어려운 결정론적 제품 가치는 무엇인가?
4. Upstage가 장식이 아니라 결과 품질에 실제로 기여하는가?
5. 사용자 효과와 서비스 논리를 어떤 증거로 보일 것인가?
6. 데모에서 전후 비교, 실패 처리, 결과 근거를 어떻게 증명할 것인가?

루브릭을 이유로 정확성·개인정보·데이터 안전선·사용자 요구를 희생하지 않는다. 유효 조합은 결정론적
엔진이 만들고, Upstage는 학사문서·강의계획서 구조화와 검증된 결과의 근거 설명에 사용한다.

## 3. 개발 워크플로우 — 코드 작성 시 반드시 따를 것

### 3-1. 작업 단위
- **한 번에 거대한 코드를 쏟지 않는다.** 동작 확인 가능한 작은 단위로 쪼갠다.
- 각 단위마다 "무엇을 확인해야 하는지"를 명시한다.
- 복붙 즉시 실행 가능한 완결 형태로 만든다.

### 3-2. 코드 품질 게이트 (자동)
코드를 작성/수정한 뒤 **커밋 전에 반드시** 아래를 통과해야 한다:

**TypeScript (web/):**
```bash
cd web && npm run lint && npm run typecheck && npm run test
```

**Python (scraper/):**
```bash
cd scraper && ruff check . && ruff format --check . && pytest
```

- 이 명령들은 **pre-commit 훅**으로 자동 실행된다 (커밋 시 자동 검사).
- **CI gate**(GitHub Actions)가 push 시 다시 검사한다. 통과 못 하면 머지 불가.

### 3-3. 피드백 루프 (린터가 오류를 잡으면)
1. 린터/타입체커/테스트 오류를 **읽는다** (무시하고 우회하지 않는다).
2. **근본 원인**을 파악한다 (오류 메시지를 억지로 눌러 끄지 않는다).
3. 코드를 고친다 → 다시 3-2 게이트를 돌린다 → 통과할 때까지 반복.
4. 같은 오류가 반복되면, 그건 **규칙(이 파일)이 부족하다는 신호**다.
   → 5장 "규칙 보강" 절차를 따라 이 파일에 규칙을 추가한다.

### 3-4. 커밋 규칙
- 커밋 메시지 말머리: `feat:` `fix:` `docs:` `refactor:` `test:` `chore:`
- 무엇을 바꿨는지 구체적으로 ("first commit" 금지).
- **API 키·비밀정보는 절대 커밋하지 않는다** (.gitignore로 제외됨. 5장 보안 참조).

## 4. 코딩 규칙 (린터가 다 못 잡는 것)

### 공통
- **API 키 하드코딩 절대 금지.** 환경변수(`UPSTAGE_API_KEY`)로만. 클라이언트에 노출 금지.
- 개인정보·전화번호를 화면·로그·커밋에 남기지 않는다.
- 주석은 "왜"를 설명한다 (무엇을은 코드가 말한다).

### TypeScript (web/)
- 파일·폴더명: 컴포넌트는 PascalCase, 그 외 kebab-case.
- `any` 지양. 불가피하면 이유를 주석으로.
- 스타일 값은 CSS 변수/Tailwind 토큰으로. 매직넘버 하드코딩 지양.
- API Route에서만 Upstage를 호출한다. **클라이언트 컴포넌트에서 직접 Upstage 호출 금지** (키 노출).

### Python (scraper/)
- 타입 힌트를 붙인다.
- 성대 API 호출 시 **요청 간격을 둔다** (서버 부담 회피 — 규칙 04-데이터안전선).
- 수집한 데이터를 **통째로 상시 재게시하지 않는다** (잡코리아 판례 리스크. docs/04 참조).
- 함수는 한 가지 일만. 수집/파싱/저장을 분리한다.

### 데이터 안전선 (법적 — docs/04에서 가져옴, 위반 시 대회·법적 리스크)
- 로그인 없는 공개 페이지만 조회 (성대 개설강좌 = 로그인 불필요, 안전).
- 전 과목 DB 통째 미러링 후 상시 게재 금지 → "사용자가 조회한 범위만 세션 단위 처리" 지향.
- 강의평(에브리타임 등) 크롤링 금지.

## 5. 자기개선 루프 (AI가 스스로 규칙을 보강·청소)

이 프로젝트는 AI가 코딩하며 **규칙을 스스로 다듬는다.** 아래 3가지 절차를 지킨다.

### 5-1. 규칙 보강 (같은 실수가 반복될 때)
- 린터/리뷰에서 **같은 종류 오류가 2번 이상** 나오면, 그건 규칙 공백이다.
- 이 파일의 4장에 규칙을 추가하고, 가능하면 **린터 설정(자동 검출)으로 승격**한다.
- 규칙 변경은 `docs/01_의사결정_로그.md`에도 한 줄 남긴다.

### 5-2. 코드 리뷰 (구현 후 자가 리뷰)
- 기능 구현 후, 다른 도구(Codex ↔ Claude Code)로 **교차 리뷰**하면 좋다.
  예: Claude Code가 짠 걸 Codex에게 "review this diff"로 검토.
- 리뷰 관점: 위 4장 규칙 위반 / 보안(키 노출) / Upstage 필연성 훼손 여부 / 죽은 코드.

### 5-3. 가비지 컬렉션 (주기적 청소) — GARBAGE_COLLECTION.md 참조
AI는 기존 코드를 보고 따라 하므로 **나쁜 패턴이 복리로 불어난다.** 주기적으로 청소한다.
- 실행 시점: 큰 기능 완료 후, 또는 주 1회.
- 점검 항목: ① 문서(docs/AGENTS)와 실제 코드가 어긋났는가 ② 규칙 위반 코드가 생겼는가
  ③ 사용하지 않는(죽은) 코드·파일·의존성이 쌓였는가 ④ 중복 로직이 여러 곳에 복사됐는가.
- 상세 절차는 `docs/GARBAGE_COLLECTION.md`에 있다.

## 6. 두 도구 병행 사용 (Claude Code + Codex)

- 둘 다 **같은 프로젝트 폴더**에서 작업한다 (VS Code 터미널에서 각각 실행).
