# 데이터 플로우 전체 정리

최종 갱신: 2026-07-23

이 서비스가 다루는 데이터를 전부 훑어서, 각각 **어디서 가져오고(추출) → 어떻게 다듬고(처리) →
어디에 쓰이는지(활용)**, 그리고 그 과정에서 실제로 겪은 시행착오를 정리한다. 사용자가 원래
꼽은 4개(학과명·개설과목·수강취득현황·졸업요건충족현황) 외에, 코드를 전수 조사해서 추가로
발견한 4개(에타 강의평 연결·친구 시간표 공유·AI 시간표 추천·브라우저 자동저장)까지 총 8개다.

## 한눈에 보기

| # | 데이터 | 원천 | 확보 방식 | 저장 위치 |
|---|---|---|---|---|
| 1 | 학과명 목록 | 성대 GAIA `selectBizType04.do` | 주기적 재수집 → 코드에 정적 시드 | `web/src/lib/skku-departments.ts` (빌드에 포함) |
| 2 | 개설과목(전공·교양) | 성대 GAIA Nexacro API | 사용자 조회 시점에 서버가 실시간 대리 호출 | 저장 안 함(매 요청 즉시 반환) + 단기 캐시 |
| 3 | 수강/취득과목 | 사용자가 올리는 GLS PDF/이미지 | Upstage Document Parse + Solar | 브라우저 메모리(세션 한정) |
| 4 | 졸업요건충족현황 | 사용자가 올리는 GLS 캡처/PDF | Upstage Document Parse + Solar, 또는 수동 템플릿 입력 | 브라우저 메모리(세션 한정) |
| 5 | 에타 강의평 연결 | 에타 검색 페이지 URL(공개) + 사용자 자신의 로그인 세션(확장, 선택) | 기본은 URL 직접 연결, 확장 설치 시 정확 매칭 보강 | 확장 로컬 캐시(우리 서버 경유 안 함) |
| 6 | 친구 시간표 공유 | 사용자가 직접 만든 시간표 | 서버가 8자리 코드로 발급·저장 | Vercel Blob(비공개) |
| 7 | AI 시간표 추천 | 후보 조합 + 사용자 성향 가중치 | 코드가 점수 계산 → Solar는 설명만 | 응답으로만 반환, 저장 안 함 |
| 8 | 브라우저 자동저장 | 사용자의 입력·선택 자체 | 클라이언트 localStorage | 이 브라우저에만(로그인 없음) |

## 1. 학과명 목록

### 사례 분석
학과 선택 드롭다운(STEP 1)에 뜨는 "경영학과", "글로벌바이오메디컬공학과" 같은 목록. 처음엔
110개짜리 정적 스냅샷을 코드에 박아 놨었는데, 사용자가 "개설강좌 사이트랑 우리 목록이 다르다"고
지적해서 실측했더니 **130개**가 맞았다 — 동아시아학술원 소속 전공 전체와 최근 신설된 융합·연계전공
다수가 통째로 빠져 있었다.

### 추출 방식
성대 GAIA 시스템의 `POST /gaia/E_NHSSU900020M/selectBizType04.do`를 호출하면 대학코드
(`TONG_HAKBU`, 4자리)별로 소속 학과 목록을 SSV 포맷으로 돌려준다. 대학코드 22개를 순회하며 전
학과를 수집한다. 로그인·쿠키 불필요.

### 처리 방법
`scraper/skku_scraper/codes.py`의 `load_departments()`가 22개 대학코드를 순회 호출해서 결과를
합치고, 학과코드/학과명/소속대학 3개 필드만 남긴 텍스트로 저장한다. 서비스 코드
(`web/src/lib/skku-departments.ts`)는 이 결과를 **빌드 시점 정적 시드**로 내장한다 — 매 요청마다
학과 목록 API를 부르지 않고, 목록이 바뀌었을 때만 재수집·재배포한다.

### 활용 방법
STEP 1 학과 검색 콤보박스, 복수전공·연계전공 검색, 그리고 개설과목 조회 시 `HAKGWA_CD` 파라미터로
쓰인다. 학과코드는 졸업요건 scope 분류(`primary_major` 등)에도 간접적으로 연결된다.

### 시행착오
- **1차 시도**: 파일로 저장하는 과정에서 132개 중 126개만 기록되는 버그가 있었다(6개 누락). 그래서
  이 파일 자체를 "진실의 원천"으로 삼지 않고, 수집기가 API를 직접 다시 호출하는 것을 원칙으로 정했다.
- **2차 발견(2026-07-18)**: 위 방식으로 재수집했더니 110→130개로 22개가 늘었다. 동아시아학술원
  전체와 성균융합원·소프트웨어융합대학 융합/연계전공 다수가 빠져 있었던 것.
- **부수 버그 동시 발견**: 새로 채워진 학과 중 다수가 4자리·8자리 학과코드였는데, 검증 로직 3곳이
  전부 "6자리 고정"으로 하드코딩돼 있었다. 목록에 넣어도 실제 선택하면 조회가 막혔을 잠재 버그를
  실제 발생 전에 코드 리뷰로 찾아서 같이 고쳤다.

## 2. 개설과목 (전공·교양)

### 사례 분석
STEP 3 "과목 담기"에서 실제로 보이는 분반 목록. 예: 경영학과·2026학년도 2학기·인문사회캠퍼스로
조회하면 214개 행(전공), 교양 영역 `GEDG001`(의사소통) 하나만 조회해도 41~44개 분반이 나온다.
넥사크로(Nexacro) 기반 SPA라 겉보기엔 JSON API가 없고, "조회 버튼을 눌러야 나오는" 화면이다.

### 추출 방식
겉으로 안 보이는 내부 API를 실제로 찾아냈다. 요약:
- 전공: `POST /gaia/E_NHSSU900020M/selectMain.do` (파라미터: 년도·학기·학과코드·캠퍼스)
- 교양: 3단계 — `selectMain01`(영역별 개설수) → `selectMain02`(영역 안 과목 목록) →
  `selectMain03`(과목의 실제 분반)
- 응답은 JSON이 아니라 **SSV**(Nexacro 자체 구분자 포맷, 레코드 구분자 `0x1e`)라서 자체 파서가 필요.
- 로그인 불필요하지만 **세션은 필요** — 매 조회 전에 먼저 `sessionLogin.do`를 호출해 세션 쿠키를
  받아야 데이터가 실제로 채워진다.

Next.js 서버(`web/src/lib/skku-course-api.ts`)가 사용자 조회 요청이 올 때마다 이 흐름을 그대로
대리 호출한다 — 별도 배치 수집이나 DB 없이 항상 실시간이다.

### 처리 방법
SSV 응답을 컬럼 스키마 → 행 데이터 순으로 파싱해서 학수번호-분반/과목명/교수/시간/캠퍼스/수업형태
같은 필드로 변환한다. 학과-캠퍼스 짝이 맞아야 하고(예: 건축학과는 자연과학캠퍼스), 없는
학과코드를 넣으면 에러가 아니라 조용히 0건이 온다는 걸 감안해 상위 계층에서 "조건을 확인해
주세요" 같은 명확한 오류로 바꿔 보여준다. 반복 조회 성능을 위해 짧은 TTL의 Blob 캐시
(`blob-cache-store.ts`)를 한 겹 얹었다.

### 활용 방법
STEP 3의 전공·교양 과목 목록, 시간표 조합 엔진의 원재료(`Subject`/`Section`), AI 필러가 미충족
교양 영역에 맞는 과목을 고를 때의 후보 풀로 쓰인다.

### 시행착오
실제로 겪은 순서대로:
1. **SSV 요청 바디를 손으로 만들어 호출 → `ErrorCode:-1`.** 넥사크로 SSV 규약을 부정확하게
   재현한 게 원인. 실제 엔진이 보내는 요청을 통신 후킹으로 관찰해서 정확한 레시피를 확보했다.
2. **`ROAD_MAP=%`(와일드카드)가 `%25`로 URL 인코딩되면 조건이 하나도 안 맞아 0행이 나온다.** 요청
   라이브러리가 자동으로 폼 인코딩을 하면서 `%`를 깨뜨렸던 것 — UTF-8 raw bytes로 직접 인코딩해서
   보내는 방식으로 해결.
3. **학과-캠퍼스 짝이 안 맞으면 0행.** 건축학과(자연과학캠퍼스 소속)를 인문사회캠퍼스로 조회하면
   1행만 나온다. 학과별 소속 캠퍼스를 정확히 매핑해야 했다.
4. **`ErrorCode:0`(성공)인데 데이터가 0행인 진짜 원인은 세션이었다.** 실제 페이지는 조회 전에
   `sessionLogin.do`를 먼저 호출해 세션을 성립시킨다. curl/Python 스크립트가 이 단계를 건너뛰면
   "성공"인데 빈 응답만 온다 — 브라우저 성능 로그로 실제 호출 순서를 재현해서 발견했다.
5. **헤더 하나(`X-NX-Content-Type: 2`)가 유일한 필수 헤더였다.** 값이 "1"이나 "0"이면, 혹은 헤더
   자체가 없으면 조용히 0행. Content-Type이나 다른 헤더는 뭘 넣어도 상관없었다 — 브라우저에서
   헤더를 하나씩 바꿔가며 실측해서 이 사실을 확정했다.
6. **교양 "전체" 조회가 실제로는 3단계 구조였다.** `HAKSU_NO=%`로 한 번에 전체를 받을 수 있을
   줄 알았는데 안 됐고, 공식 화면과 똑같이 영역 조회 → 과목 조회 → 분반 조회 3단계를 순서대로
   거쳐야 했다.
7. **교양 영역 코드 하나가 틀려 있었다.** "미래(SW/AI)" 영역 코드를 `A3`으로 추정했었는데, 실제
   화면 스크립트를 확인해보니 `A7`이었다.

## 3. 수강/취득과목

### 사례 분석
사용자가 올리는 GLS "개인별 수강/취득 과목 리스트" PDF. 실제 문서 기준 열 구성은 전공/이수구분/
년도/학기/학수번호+과목명/영역/학점/성적/비고. 실측 문서 하나(33과목, 학점 합계 94.0)로 여러
차례 재현 테스트를 했다.

### 추출 방식
서버 API(`POST /api/parse-academic-document`, `kind=course_history`)가 업로드된 PDF/이미지를
Upstage **Document Parse**로 표 구조 텍스트(HTML+Markdown)로 변환한 뒤, 그 결과를 Upstage
**Solar**(구조화 출력 JSON Schema)에 넘겨 고정된 필드로 정리한다.

### 처리 방법
"표 파서(코드, 결정론적) + Solar(AI, 보완)" 하이브리드다. 학수번호는 `[A-Z]{2,6}[0-9]{3,4}`
정규식으로 검증하고, `취득학점:`/`합계:` 같은 요약 행은 과목으로 만들지 않으며, 한 셀에 과목
2개가 붙어 있으면 각각의 draft로 분리한다. **표 파서가 코드·과목명을 정확히 읽은 행은 Solar가
뭘 반환하든 버리지 않고**, Solar 값은 전공범위·이수구분 같은 부가정보 보강에만 쓴다. 정확한
성적 등급은 추천에 불필요하므로 구조화 직후 폐기하고, 이수/실패/철회 여부만 남긴다.

### 활용 방법
`completionStatus=이수`이고 재수강 토글을 켜지 않은 학수번호는 STEP 3 후보 목록에서 자동 제외된다.
분반 정보는 과거 이력에 없으므로 학수번호 단위까지만 비교한다.

### 시행착오
- **같은 증상(검토 필요 항목이 매번 다름)을 졸업요건과 똑같은 방식으로 고쳤는데 효과가 없었다.**
  원인을 다시 추적해보니 이쪽은 애초에 구조가 정반대였다 — 졸업요건은 "표가 주인공"이었는데
  수강/취득과목은 "Solar가 주인공, 표는 보조"로 짜여 있었다.
- **더 근본적인 버그: 표 파서가 HTML 표를 아예 못 읽고 있었다.** 졸업요건 표 파서는 마크다운·HTML
  표 둘 다 처리했지만, 이쪽은 마크다운만 처리하도록 짜여 있어서 Document Parse가 HTML로 표를
  주면 표 파서가 0행을 인식하고 조용히 포기했다 — 처방을 적용할 데이터 자체가 없었던 것. HTML
  표 파싱을 추가한 뒤에야 원래 처방이 실제로 작동했다.
- **수정 전/후 실측 비교**(동일 문서, API 3회 반복 호출): 검토 필요 이유가 "빈칸을 이유로 지어냄 →
  성적을 이유라고 반복 → 프롬프트 문구 그대로 echo"에서 → **33과목 전부, 3회 모두 완전히 빈 배열**로
  바뀌었다. 0학점 P등급 과목 하나가 "이수완료"↔"검토필요"를 오가던 것도 3회 모두 동일하게 고정됐다.
- **최근 추가 버그(2026-07)**: Solar가 만드는 `earnedCredits`(해석값)와 `rawValues.취득학점`(원문
  echo)이 서로 어긋나는 경우가 있었다(예: 원문 "9"인데 해석값 36). 표 파서가 못 잡는 행에서만
  발생 — 이제 이 둘을 서로 대조해서 다르면 원문 값을 신뢰하도록 고쳤다.
- **300초 타임아웃 버그**: 학수번호 누락 재시도 단계가 잘린 markdown 없이 전체 문서를 다시 보내서,
  큰 문서(복수전공 등)일수록 이 재시도가 더 오래 걸리는 역설이 있었다. 재시도도 표 구간만 추려서
  보내도록 고치고, Document Parse 75초·Solar 55초 개별 타임아웃도 추가했다.

## 4. 졸업요건충족현황

### 사례 분석
"영역별 학점취득/수강현황" 표(구분/기준학점/취득학점/수강학점(학기별)/잔여학점)와 "이수구분별"
표 두 종류. 균형교양처럼 "3개 영역 중 2개 이상, 합계 6학점"식 복합 규칙도 있다.

### 추출 방식
수강/취득과목과 같은 API, `kind=graduation_requirements`. 여러 장의 캡처를 누적 첨부·붙여넣기할
수 있고, 각 파일을 따로 Parse/Solar 처리한 뒤 브라우저 메모리에서 하나로 합친다.

### 처리 방법
여기는 **표가 주인공**이다 — 결정론적 표 파서가 우선이고 Solar는 표가 못 잡은 행만 보완한다.
숫자형 요건은 `기준-취득-수강중` 공식으로 잔여학점을 직접 재계산해서 문서에 적힌 잔여학점과
대조하고, 다르면 `status=review`로 표시한다(추측하지 않는다). 균형교양 같은 그룹 규칙은
`totalAreas`/`minimumAreas`/`totalCredits`로 표현해서 "두 영역 이상에서 합계 6학점"을 정확히
판정한다. C/L(공동개설) 과목이 여러 전공 취득학점에 중복 표시될 수 있어서, 요건 행끼리 취득학점을
더해 전체 취득학점을 만들지 않는다.

### 활용 방법
`scope=general`이면서 미충족인 요건 라벨이 ①교양 카탈로그 기본 탭 선택, ②AI 필러 후보 우선순위,
③AI 추천 결과의 "졸업요건 기여" 문구, ④친구 리믹스 강도 다이얼까지 총 4곳에 쓰인다(자세한 내용은
이전 대화에서 정리한 4곳 참고). 서류가 없는 사용자를 위해 15개 표준 요건 영역 템플릿을 만들어
"영역만 고르고 잔여학점만 입력"하는 수동 입력 경로도 별도로 열어놨다.

### 시행착오
- **똑같은 문서를 3번 반복 호출했는데 결과가 매번 달랐다.** "검토 필요" 항목이 0개였다가
  갑자기 많아지고, 한 번은 JSON 스키마 설명 문구(`["string"]`)를 그대로 베낀 값을, 한 번은
  프롬프트 지시문 자체를 되풀이하는 값을 내놓기도 했다 — 문서 내용과 무관한 AI의 순수 형식
  오작동이었다.
- **구조화 출력(JSON Schema strict mode) 지원 여부가 검색으로는 불확실**해서, 실제 API를 직접
  호출해 지원을 확인하고 적용했다. 적용 후 형식 오작동은 100% 사라졌지만, "몇 개 항목을 반환할지"
  같은 비일관성은 남아 있었다.
- **최종 처방**: 표 파서가 이미 정확히 처리한 행에 대해서는 Solar의 판정(검토 이유 등)을 아예
  반영하지 않고 코드가 계산한 값만 신뢰하도록 구조를 바꿨다. 수정 전/후 실측(API 3회 반복): 검토
  필요 이유가 "3회 결과가 매번 다름" → **3회 결과가 바이트 단위로 완전히 동일**, 15개 항목의 충족
  상태도 3회 모두 동일해졌다.
- **"제2/3전공" 라벨을 못 알아봐서 전부 '기타'로 분류되던 버그**(복수전공 학생 보고): 해결됨.
  `RequirementScope`에 `secondary_major`를 추가하고, Solar 응답 별칭과 표 라벨 `제2전공`·`제3전공`을
  모두 `secondary_major`로 정규화했다. `제3전공 총학점` 회귀 테스트도 포함한다.

## 5. 에브리타임 강의평 연결

### 사례 분석
"이 과목 강의평 보기" 버튼을 누르면 에브리타임의 정확한 과목·교수 리뷰 페이지로 바로 연결되는
기능. 우리 서버는 에타 데이터를 전혀 만지지 않는다.

### 추출 방식
최종 채택한 방식은 강의평 검색 페이지(`everytime.kr/lecture/search`)가 받는 URL 쿼리 파라미터
구조(`keyword`, `condition=name|professor`)다. 사용자가 직접 브라우저로 검색해 보면서 알아냈고,
브라우저가 이 페이지에 접속할 때 쓰는 평범한 GET 링크라 크롤링에 해당하지 않는다. 이 URL은
로그인 여부와 무관하게 항상 열리는 **기본(미설치) 경로**이고, 여기에 더해 사용자가 설치하는 크롬
확장(Manifest V3, `extension/`)이 **자신의 이미 로그인된 에타 세션 안에서** 같은 URL을 열어 과목명·
교수명이 정확히 일치하는 리뷰 페이지까지 자동으로 찾아 이동해 주는 **선택적 보강**을 얹는다. 우리
웹페이지는 `CustomEvent`(`skku-timetable:connector-request`/`-response`)로 확장의 콘텐츠
스크립트와 통신할 뿐, 서버는 에타 도메인에 전혀 접근하지 않는다.

### 처리 방법
과목명+교수명으로 `everytime.kr/lecture/search`를 열고(교수명이 있으면 `condition=professor`
우선, 없으면 `condition=name`), 확장이 설치돼 있으면 검색 결과에서 과목명·교수명이 모두 일치하는
항목만 최종 매칭한다(`review-match.js`). 여러 개 후보가 남으면 자동으로 추측하지 않고 사용자에게
직접 고르게 한다.

### 활용 방법
과목 선택 화면과 완성된 시간표 화면 양쪽에서 항상 접근 가능한 버튼으로 노출한다. 확장 설치 여부와
무관하게 버튼은 항상 동작하고(미설치 시 검색 결과 페이지, 설치 시 정확한 리뷰 페이지로 이동), 설치를
사용 조건으로 요구하지 않는다.

### 시행착오
- **에타 내부(비공개) API까지 실제로 찾아냈지만 정책상 채택하지 않았다.** 로그인한 세션에서
  시간표 화면의 "강의 검색"이 호출하는 요청(`POST api.everytime.kr/find/timetable/subject/list`)을
  잡아보니, 응답 XML의 각 과목 태그에 `lectureId`(강의평 페이지 ID)와 `lectureRate`(별점)가 이미
  포함돼 있었다. `https://everytime.kr/lecture/view/{lectureId}`로 링크를 바로 조합할 수 있고,
  카테고리 목록 API(243개 전공/교양 카테고리)를 페이지네이션으로 순회하면 추가 요청 없이 전체
  6,254개 강의의 링크·별점을 정확히 뽑아낼 수 있었다 — 실제로 이 방식으로 과목·교수·별점·강의평
  URL을 연결한 엑셀 파일까지 만들어봤다. 하지만 이건 명백히 "비공개 API 자동 호출"이라 에타 이용약관
  위반 소지가 커서, 서비스에는 채택하지 않고 폐기했다.
- 이 판단 이후 "그럼 서버가 공개 검색 페이지라도 대신 가져오면 어떨까"도 검토했지만, 이용약관의
  크롤링 금지 조항과 실제 로그인 요구(로그아웃 상태로는 검색 페이지 자체가 로그인 화면으로
  리다이렉트됨)를 확인하고 이 경로도 접었다 — 서버는 어떤 형태로든 에타에 접근하지 않는 쪽으로 확정.
- 사용자가 직접 실제 URL 패턴(`?keyword=...&condition=name|professor`)을 테스트해서 알려준
  값으로 최종 확정했다.
- 초기 버전은 과목명 검색만 지원했는데, 동명 과목이 여러 개일 때 엉뚱한 교수 리뷰로 갈 위험이
  있어서 교수명 우선 검색으로 바꿨다.
- 확장프로그램은 "설치해야만 정확한 리뷰 페이지로 바로 간다"는 한계가 뚜렷했다. 그래서 확장을
  핵심 경로로 삼지 않고, 설치 없이도 항상 동작하는 검색 페이지 연결을 기본값으로 두고 확장은
  그 위에 얹는 선택적 보너스로만 남겼다 — 설치를 사용 조건으로 강제하지 않기 위한 설계 결정이었다.

## 6. 친구 시간표 공유

### 사례 분석
"코드로 공유" 버튼을 누르면 8자리 코드(`23456789ABCDEFGHJKMNPQRSTUVWXYZ` 알파벳, 헷갈리는
0/O/1/I/L 제외)가 생성된다. 친구는 이 코드만 입력하면 로그인 없이 시간표를 볼 수 있다.

### 추출 방식(=저장 방식)
사용자가 "저장"을 누르는 시점에만 서버가 관여한다. Vercel Blob(비공개 접근)에 과목 목록·소유자
표시명·수정용 토큰 해시를 JSON으로 저장한다. 코드가 없으면 새로 발급, 있으면(+토큰 검증 통과)
같은 코드에 덮어써서 "재저장해도 코드가 안 바뀌고 계속 최신화"되는 경험을 만든다.

### 처리 방법
수정 권한은 로그인이 아니라 `editToken`(발급 시 1회 전달, 이후 해시만 서버에 보관)으로 증명한다.
과목 배열은 최대 30개로 제한하고 스키마를 검증해서 저장한다.

### 활용 방법
STEP 3~4에서 만든 시간표를 저장 → 코드 공유 → `/friends`에서 여러 명 시간표 겹쳐보기, 그리고
`/friends/remix`에서 친구 시간표를 참고해 새 시간표를 만드는 기능(강도 다이얼 포함)의 입력이 된다.

### 시행착오
- **초기 저장 스키마엔 "이 과목이 필수였는지 선택이었는지" 구분이 없었다.** 그래서 리믹스 기능을
  만들 때 어떤 과목을 "내 고정 기준"으로 삼아야 할지 알 수 없었다 — `requiredCourseIds` 필드를
  추가해서 해결했는데, 기존에 이미 만들어진 공유는 이 필드가 없다.
- **레거시 공유 파싱 버그**: 위 필드 추가 후, 필드가 아예 없던 옛날 공유를 읽으면 `null`이
  정상인데 리믹스 전용 파서가 이를 "형식이 잘못됨"으로 오판해서 "시간표 응답 형식이 올바르지
  않습니다" 오류를 반복 표시했다. `undefined`/`null` 둘 다 "정상적인 레거시 상태"로 읽도록 수정.
- **친구 데이터를 어디서 읽을지 설계 이슈**: 리믹스 화면은 `/friends`와 다른 라우트라 React
  상태를 직접 공유할 수 없었다. 코드가 이미 `localStorage`에 있으니 기존 조회 API를 다시
  부르면 된다고 판단해서, 새 API나 페이지 간 상태 전달 장치를 따로 만들지 않았다.

## 7. AI 시간표 추천

### 사례 분석
STEP 5에서 "공강 희망일 3개, 연강 회피 강함" 같은 조건을 고르면 상위 시간표 몇 개와 "왜 이걸
추천하는지" 짧은 설명이 붙어서 나온다.

### 추출 방식
입력은 이미 결정론적으로 만들어진 유효 시간표 후보들(다른 데이터 처리 없이 조합 엔진 산출물)과
사용자가 STEP 5에서 고른 가중치 설정(공강/연강회피/점심시간/아침수업회피/요일몰아듣기/수업형태
선호, 각각 1~5단계 중요도)이다.

### 처리 방법
**점수 계산은 전부 코드가 한다** — `timetable-scoring.ts`가 각 후보의 원점수(raw value)를 계산해
가중치별 중요도를 곱하고 합산해서 정렬한다. 여기까지 Solar는 관여하지 않는다. 상위 N개가 정해진
뒤에야 Solar(`requestSolarCompletion`)를 호출해서 "왜 이 조합이 좋은지" 설명 문장만 붙인다.
Solar 호출이 실패해도(`aiExplanationFailed`) 이미 계산된 시간표 순위 자체는 그대로 반환한다.
"졸업요건 기여" 문구는 Solar가 아니라 코드가 후보의 실제 교양 과목과 미충족 라벨을 대조해서
만든다(4번 항목 참고) — Solar가 근거 없이 "이것도 도움된다"고 지어내는 문제를 막기 위해서다.

### 활용 방법
STEP 5 결과 화면, 그리고 친구 리믹스 화면의 유사도/차별화 점수 계산에도 같은 가중치 채점 로직
일부를 재사용한다.

### 시행착오
- 가중치 체계를 한 번 크게 갈아엎었다 — `importance`를 `low/medium/high` 3단계에서 숫자
  `1~5` 5단계로 바꾸고, `compact_days`+`minimize_daily_span`을 `day_packing`(몰아듣기/분산
  선택 가능) 하나로, `prefer_in_person`+`prefer_online`을 `course_format`(선호 형태 선택)
  하나로 합쳤다. 옵션 수를 줄이면서 표현력은 늘리는 방향이었다.
- AI 설명이 매번 다르게 나오는 문제는 다른 문서 파싱 데이터(3·4번)와 같은 계열의 이슈였지만,
  여기는 "판정"이 아니라 "설명" 역할이라 완전히 결정론적으로 만들 필요는 없다고 판단하고 그대로
  뒀다 — 대신 판정(어떤 시간표가 몇 등인지)은 처음부터 전부 코드가 맡도록 설계했다.

## 8. 브라우저 자동저장 (로그인 없는 이어하기)

### 사례 분석
"시간표 다시 짜고 싶은데 기본정보 입력부터 매번 다시 하기 귀찮다"는 사용자 피드백에서 시작한
기능. 로그인 없이, 이 브라우저에서만 이어서 작업할 수 있게 했다.

### 추출 방식(=수집 시점)
사용자가 입력하는 순간 그대로. 서버로는 전혀 전송하지 않는다.

### 처리 방법
민감도에 따라 저장 정책을 다르게 뒀다.
- **기본정보(STEP 1)·담은 과목/선택묶음/고정일정(STEP 3)**: 민감하지 않다고 판단해 **항상
  자동 저장·복원**한다. 학과·연도·학기 키로 스코핑해서 다른 조회 조건의 저장값이 섞이지 않게
  하고, 실시간 개설강좌 응답이 항상 우선하며 저장값은 로딩 중이거나 조회 실패했을 때만 빈틈을
  메운다.
- **분석결과(STEP 2, 수강내역·졸업요건)**: 학점·이수현황처럼 민감할 수 있어서 기본값은 그대로
  "새로고침하면 삭제"를 유지한다. 사용자가 "이 브라우저에 분석 결과 보관하기"를 직접 켰을 때만
  구조화된 결과(JSON)를 저장한다. 원본 PDF/이미지 바이트는 절대 저장하지 않고, 언제든 "보관한
  분석 결과 삭제" 버튼으로 지울 수 있다.

모든 저장·복원 함수는 저장된 데이터의 형식을 엄격히 검증하고, 손상되거나 형식이 안 맞으면
조용히 무시한다(잘못된 데이터로 화면을 망가뜨리지 않기 위해).

### 활용 방법
STEP 1~3 전체에 걸쳐 "브라우저를 닫았다 열어도 이어서 할 수 있다"는 경험을 만든다. 서버 저장이나
계정 생성 없이 순수 클라이언트 기능이다.

### 시행착오
- **기본정보가 복원됐는데 입력창엔 placeholder만 보이는 버그**: 저장소에서 프로필이 복원되는
  시점이 첫 렌더 이후라, 입력창 자체의 로컬 검색 상태는 그 시점에 비어 있었다. 목록이 닫힌
  상태에서는 확정된 학과명을 우선 표시하도록 고쳤다.
- **STEP 3 과목 목록 스크롤이 계속 맨 위로 튀는 버그**: 필수 과목 체크 → 자동저장(localStorage
  기록) → 부모 컴포넌트 리렌더 → 저장된 계획 객체가 새로 생성 → 그런데 강좌 조회 `useEffect`가
  이 저장 계획까지 의존성으로 물고 있어서, 체크 한 번에도 목록을 통째로 비우고 다시 불러오면서
  스크롤이 초기화됐다. 강좌 조회는 학과·학기 조건에만 반응하도록, 저장 계획 복원은 조회당 한
  번만 실행되는 별도 effect로 분리해서 해결.
- 도입 전 검토: `sessionStorage`(새로고침엔 살아남고 탭을 닫으면 사라짐)도 후보였지만, STEP 2
  분석결과는 이미 "새로고침하면 사라진다"는 개인정보 안내를 명시적으로 해둔 상태라 그 약속과
  충돌한다고 판단해 제외했다. 대신 옵트인 토글이 있는 `localStorage`로 최종 결정했다.

## 공통으로 반복된 패턴

세 부류의 데이터(2·3·4번)에서 독립적으로 같은 결론에 도달했다: **AI(Solar)에게 판단을 맡기는
범위를 최대한 줄이고, 코드가 확실히 계산할 수 있는 부분은 코드가 담당한다.** 잔여학점 재계산,
시간표 점수 산정, 졸업요건 기여 문구 계산이 전부 이 원칙을 따른다. Solar는 "판정"이 아니라
"설명·해석"에만 남아 있다. 이건 중간점검 멘토링에서 받은 조언("계산은 코드로, AI는 설명·해석만")과
독립적으로 실측을 통해 재확인한 결론이기도 하다.

두 번째 패턴: **"체감상 이상하다"는 보고를 감으로 고치지 않고, 실제 API를 여러 번 반복 호출해
재현·정량화한 뒤 처방하고, 처방 후에도 같은 방식으로 재검증했다.** 3·4번 항목의 "3회 반복 호출
결과 비교표"가 그 증거다.
