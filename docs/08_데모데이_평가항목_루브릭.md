## 3. 이 프로젝트의 평가 스토리

### 문제와 타겟

- 타겟: 수강 이력, 졸업요건, 개설강좌, 강의계획서가 흩어져 있어 다음 학기 시간표를 직접 대조해야 하는 성균관대 학부생.
- 문제: 범용 챗봇에 문서를 한 번 넣는 것만으로는 최신 개설강좌, 분반, 시간 충돌, 재수강 여부, 졸업요건, 사용자 제약을 모두 만족하는 누락 없는 조합을 보장하기 어렵다.
- 결과: 사용자가 확인한 학사정보와 공식 개설강좌로 결정론적 유효 시간표를 만들고, Upstage가 문서를 구조화하고 근거를 설명한다.

### 범용 LLM과의 차이

이 서비스는 단발성 대화가 아니라 다음을 재현 가능한 파이프라인으로 묶는다.

1. 개인별 수강/취득과목과 졸업요건 문서를 구조화한다.
2. 사용자가 자동 추출 결과를 검토·수정·확정한다.
3. 최신 공식 개설강좌를 선택 범위에서 조회한다.
4. 기수강 제외, 재수강 예외, 분반, 선택 그룹, 학점 범위, 시간 충돌과 사용자 제약을 결정론적으로 계산한다.
5. 유효성이 보장된 조합만 시각화하고, Upstage가 문서 근거를 활용해 설명한다.

핵심 가치는 공식 데이터 연결, 사용자 승인, 완전한 경우의 수 모델, 제약 검증, 출처와 상태가 남는 구조에 있다. AI가 유효 조합을 임의로 만들어서는 안 된다.

### 정보 처리 파이프라인

```text
사용자 학사문서(PDF/이미지/캡처)
  → Upstage Document Parse
  → Solar 정규화
  → 서버 스키마 검증·보완
  → 사용자 검토·확정
  → 성균관대 공식 개설강좌 조회
  → 기수강/재수강·과목/분반·선택 그룹·학점·시간 제약 결합
  → 결정론적 유효 시간표 생성
  → 강의계획서 Parse와 Solar를 이용한 근거 설명
  → 시각 시간표와 확인 가능한 결과
```

### Upstage의 필연적 역할

- Document Parse: 사람이 다시 입력하기 어려운 학사문서와 강의계획서의 표·문장을 구조화한다.
- Solar: Parse 결과를 고정 스키마로 정규화하고, 검증된 시간표에 문서 기반 설명을 붙인다.
- 서버 검증과 사용자 확인: AI 추출을 무조건 신뢰하지 않고 오류를 드러내며 수정할 수 있게 한다.
- 데모에서는 Upstage를 뺐을 때 무엇이 수작업으로 되돌아가고 어떤 결과가 사라지는지 보여준다.

## 4. AI 개발 도구의 필수 판단 게이트

모든 큰 기능, 데이터 구조 변경, 우선순위 결정, 데모 준비 전에 아래 질문에 답한다.

1. 어떤 흩어진 정보를 어디서, 왜 가져오는가?
2. 입력이 어떤 검증·변환 단계를 거쳐 사용자 결과가 되는가?
3. 범용 LLM만으로 대체하기 어려운 결정론적·제품적 가치는 무엇인가?
4. Upstage가 장식이 아니라 결과 품질에 기여하는 지점은 무엇인가?
5. 사용자가 얻는 시간 절약, 오류 감소, 의사결정 개선은 무엇인가?
6. 데모에서 이를 어떤 화면, 수치, 전후 비교, 실패 처리로 증명할 것인가?

답이 불분명하면 사용자 가치와 평가 기여를 다시 정의한 뒤 착수한다. 루브릭은 점수 장식이 아니라 제품의 논리·정확성·증거를 강화하는 우선순위 기준이다.

### 구현 시 지켜야 할 원칙

- 유효 시간표 생성은 결정론적 엔진이 담당한다. Solar가 존재하지 않는 조합을 만들거나 충돌 검증을 대신하지 않는다.
- AI 추출은 초안이며 서버 검증과 사용자 확인을 거친다.
- 데이터 출처, 변환 단계, 검토 상태와 실패 사유를 숨기지 않는다.
- Upstage 호출 전후의 기여를 테스트·샘플·화면으로 남긴다.
- 기능 수보다 파이프라인 완결성, 예외 처리, 재현성과 실제 사용성을 우선한다.
- 부담도 점수화·상위 N 추천은 별도 검증 전까지 구현하지 않는 D-13을 유지한다.

## 5. 데모·발표 증거 체크리스트

- [ ] 수강/취득과목 원본이 구조화되고 사용자가 검토·확정하는 장면
- [ ] 졸업요건 캡처 붙여넣기 → Parse/Solar → 복합 규칙 검토 장면
- [ ] 공식 개설강좌와 개인 이력이 결합되어 기수강 과목이 제외되는 장면
- [ ] 여러 분반과 선택 그룹으로 누락 없는 경우의 수를 만드는 장면
- [ ] 시간 충돌·사용자 제약·학점 범위를 만족하는 조합만 남는 장면
- [ ] 범용 챗봇 단발 답변과 검증 가능한 서비스 결과의 차이
- [ ] Upstage 적용 전후의 수작업량 또는 구조화 품질 비교
- [ ] AI 추출 실패·불확실 항목을 검토로 보내는 장면
- [ ] 시간 절약·오류 감소 등 최소 한 가지 사용자 근거
- [ ] 입력 → 처리 → 검증 → 결과를 보여주는 발표용 파이프라인 도식
