[D-01] 아이디어 발산 방식 = 생성×선택 분리
- 결정: 단일 최선안 대신 겹치지 않는 후보를 다수 발산 후, novelty/value 2축 분리 채점으로 선택.
- 이유: 프로젝트 지침(창의성=생성×선택). 발산 단계 자기검열 금지.

[D-02] 조원 아이디어 4개 중 3개 드롭
- 드롭: 구직 크롤링 원안 / 리얼프라이스(뷰티 최저가) / 디택(직구 원본매칭)
- 이유: 공통적으로 ① Upstage 미사용(Claude API 씀) ② 크롤링/웹검색이 서비스 심장
  → 대회 필수요건("Parse/Extract/Solar 중 1개 필연 사용") 미충족.
- 부분 생존: 구직案의 "이력서 vs 공고 갭분석"은 업로드 기반으로 재설계 시 유효.

[D-03] 조원 아이디어 매치유는 재설계 시 생존
- 결정: 교환학생 매칭(매치유)은 살릴 가치 큼. 단 "웹검색 의존 → 문서 업로드+Parse"로 전환 조건.
- 이유: 3축 융합(전공×예산×성향)은 심사 상단 기대치에 맞고 novelty 높음.

[D-04] 메인 후보 = 시간표 조합 추천
- 결정: 재성이 제안한 시간표 아이디어를 메인으로 채택.
- 이유: 흩어진 정보/데이터 융합/Action/챗봇 차별화/Upstage 필연성 5개 축 모두 충족하는
  드문 후보. novelty·value 동시에 높음.
- 근거 축적: "에타조차 전국 시간표 자동화 못함(400개중 109개만 지원)" = 검증된 페인포인트.

[D-05] 데이터 확보 방식 = 크롤링 대신 우선 "업로드 기반" 검토 → 이후 공개API 발견으로 조정
- 초기 결정: 약관 리스크 회피 위해 강의계획서 PDF 업로드 방식 우선.
- 조정: 성대 개설강좌가 로그인 없이 열리는 공개 API임을 발견 → 수집도 옵션에 포함.
  (상세 경위: 02_기술검증_기록)

[D-06] 서비스 범위 = 성균관대 1개교 한정
- 결정: 전국 확장 아니라 성대만 타깃.
- 이유: 단국대 선례(황준일)도 "전국 자동화 무리 → 모교만"으로 성공. 완성도·진정성에서 오히려 유리.
- 발표 프레이밍: "성대 파일럿 → 구조 확장 가능"으로 말한다.
  (특히 PDF 업로드 방식이면 학교 독립적 파이프라인이라 확장논리가 거짓이 아님)

[D-07] MVP 메뉴 = 전공 + 교양 2개만
- 결정: 대학원·도전학기·ESG 제외.
- 이유: F12 조사 결과 메뉴마다 조회조건이 전부 다름(하나의 로직으로 못 긁음).
  학부생 시간표엔 전공+교양이면 충분. 개발부담 감소.

[D-08] 평점 데이터 = 크롤링 금지, 우회 확보
- 결정: 에브리타임 등 강의평 크롤링은 안 한다(약관위반·저작물).
- 우회안: A) 성대 성적분포 공시자료 있으면 Parse / B) 사용자 직접 입력 / C) v1은 평점 빼고 시간+교수만.
- 상태: 성대 공시 여부 미확인 → 05_미해결_과제 플래그.

[D-09] 데이터 수집 방법 A/B/C — 미확정 (검토중) → [번복됨: D-11]
- A. 순수 API 직접호출(Python requests): 단국대式. 최고 속도. 단 넥사크로 SSV/XML body 정확 재현 필요.
- B. 브라우저 자동화+렌더링 추출: 검증완료·안정적이나 느리고 무거움.
- C. 넥사크로 엔진 위임(transaction 직접호출): SSV 손으로 안 만들고 엔진이 파싱까지. 가장 빠를 것으로 추정.
- 잠정 판단: "사용자 조회 시점 처리"면 B로 충분. "전과목 상시수집"이면 C/A 필요.
  법적 안전선(DB통째복제 금지) 고려 시 조회시점 처리(B)가 무난.
- 상태: C안 실제 검증은 아직 안 함 → 05_미해결_과제.

[D-11] 데이터 수집 방법 = A안(순수 API 직접호출) 확정 (2026-07-11)
- 결정: A안으로 간다. B는 폴백.
- 경위: C안(넥사크로 엔진 통신 후킹)으로 엔진이 실제 보내는 요청 전문을 관찰 →
  A안의 완전한 레시피(엔드포인트·헤더·SSV 파라미터·응답 스키마) 확보. (상세: 02_기술검증_기록 방법A)
- 이유: A안은 서버/GitHub Actions에서 독립 실행 가능(단국대式) + 빠름. 초기 A 실패는 SSV 재현 미숙
  때문이었고, 이제 정답 레시피가 있으니 해결됨.
- 재확인된 사실: 메뉴마다 엔드포인트 다름 (전공=selectMain.do/E_NHSSU900020M,
  교양=selectMain03.do/E_NHSSU900010M) → 전공+교양 2메뉴 한정(D-07) 결정 더욱 타당.
- 개발 전 확인: 학과코드(HAKGWA_CD) 목록 / 쿠키 필수 여부 / TERM 코드 매핑 → 05_미해결_과제 P3.

[D-10] ★ 핵심 방향전환 검토 = 옵션 A (강의계획서 파싱 중심 재설계)
- 문제인식: 현재 시간표 구도는 크롤링+조합계산이 심장이고 Upstage가 곁가지 → 조원 드롭案과 같은 함정.
- 권장방향(옵션A): 무게중심을 강의계획서 PDF 파싱으로 이동.
  개설강좌 수집=최소 시딩, 진짜가치=강의계획서 Parse/Extract(평가방식·과제비중·시험·수업유형).
  → "강의계획서를 읽어 학기 부담까지 예측하는 시간표"로 재프레이밍.
- 대안: 옵션B(다른 후보 선회) / 옵션C(현구도+프레이밍 방어, 리스크 큼).
- 상태: ★ 팀 회의에서 결론낼 최우선 안건. 미확정.
