[D-01] 아이디어 발산 방식 = 생성×선택 분리
- 결정: 단일 최선안 대신 겹치지 않는 후보를 다수 발산 후, novelty/value 2축 분리 채점으로 선택.
- 이유: 프로젝트 지침(창의성=생성×선택). 발산 단계 자기검열 금지.

[D-02] 조원 아이디어 4개 중 3개 드롭
- 드롭: 구직 크롤링 원안 / 리얼프라이스(뷰티 최저가) / 디택(직구 원본매칭)
- 이유: 공통적으로 ① Upstage 미사용(Claude API 씀) ② 크롤링/웹검색이 서비스 심장
  → 대회 필수요건("Parse/Extract/Solar 중 1개 필연 사용") 미충족.
- 부분 생존: 구직案의 "이력서 vs 공고 갭분석"은 업로드 기반으로 재설계 시 유효.

[D-03] 조원 아이디어 매치유는 재설계 시 생존
- 결정: 교환학생 매칭(매치유)은 살릴 가치 큼. 단 "웹검색 의존 → 문서 업로드+Parse"로 전환 조건.
- 이유: 3축 융합(전공×예산×성향)은 심사 상단 기대치에 맞고 novelty 높음.

[D-04] 메인 후보 = 시간표 조합 추천
- 결정: 재성이 제안한 시간표 아이디어를 메인으로 채택.
- 이유: 흩어진 정보/데이터 융합/Action/챗봇 차별화/Upstage 필연성 5개 축 모두 충족하는
  드문 후보. novelty·value 동시에 높음.
- 근거 축적: "에타조차 전국 시간표 자동화 못함(400개중 109개만 지원)" = 검증된 페인포인트.

[D-05] 데이터 확보 방식 = 크롤링 대신 우선 "업로드 기반" 검토 → 이후 공개API 발견으로 조정
- 초기 결정: 약관 리스크 회피 위해 강의계획서 PDF 업로드 방식 우선.
- 조정: 성대 개설강좌가 로그인 없이 열리는 공개 API임을 발견 → 수집도 옵션에 포함.
  (상세 경위: 02_기술검증_기록)

[D-06] 서비스 범위 = 성균관대 1개교 한정
- 결정: 전국 확장 아니라 성대만 타깃.
- 이유: 단국대 선례(황준일)도 "전국 자동화 무리 → 모교만"으로 성공. 완성도·진정성에서 오히려 유리.
- 발표 프레이밍: "성대 파일럿 → 구조 확장 가능"으로 말한다.
  (특히 PDF 업로드 방식이면 학교 독립적 파이프라인이라 확장논리가 거짓이 아님)

[D-07] MVP 메뉴 = 전공 + 교양 2개만
- 결정: 대학원·도전학기·ESG 제외.
- 이유: F12 조사 결과 메뉴마다 조회조건이 전부 다름(하나의 로직으로 못 긁음).
  학부생 시간표엔 전공+교양이면 충분. 개발부담 감소.

[D-08] 평점 데이터 = 크롤링 금지, 우회 확보
- 결정: 에브리타임 등 강의평 크롤링은 안 한다(약관위반·저작물).
- 우회안: A) 성대 성적분포 공시자료 있으면 Parse / B) 사용자 직접 입력 / C) v1은 평점 빼고 시간+교수만.
- 상태: 성대 공시 여부 미확인 → 05_미해결_과제 플래그.

[D-09] 데이터 수집 방법 A/B/C — 미확정 (검토중) → [번복됨: D-11]
- A. 순수 API 직접호출(Python requests): 단국대式. 최고 속도. 단 넥사크로 SSV/XML body 정확 재현 필요.
- B. 브라우저 자동화+렌더링 추출: 검증완료·안정적이나 느리고 무거움.
- C. 넥사크로 엔진 위임(transaction 직접호출): SSV 손으로 안 만들고 엔진이 파싱까지. 가장 빠를 것으로 추정.
- 잠정 판단: "사용자 조회 시점 처리"면 B로 충분. "전과목 상시수집"이면 C/A 필요.
  법적 안전선(DB통째복제 금지) 고려 시 조회시점 처리(B)가 무난.
- 상태: C안 실제 검증은 아직 안 함 → 05_미해결_과제.

[D-11] 데이터 수집 방법 = A안(순수 API 직접호출) 확정 (2026-07-11)
- 결정: A안으로 간다. B는 폴백.
- 경위: C안(넥사크로 엔진 통신 후킹)으로 엔진이 실제 보내는 요청 전문을 관찰 →
  A안의 완전한 레시피(엔드포인트·헤더·SSV 파라미터·응답 스키마) 확보. (상세: 02_기술검증_기록 방법A)
- 이유: A안은 서버/GitHub Actions에서 독립 실행 가능(단국대式) + 빠름. 초기 A 실패는 SSV 재현 미숙
  때문이었고, 이제 정답 레시피가 있으니 해결됨.
- 재확인된 사실: 메뉴마다 엔드포인트 다름 (전공=selectMain.do/E_NHSSU900020M,
  교양=selectMain03.do/E_NHSSU900010M) → 전공+교양 2메뉴 한정(D-07) 결정 더욱 타당.
- 개발 전 확인: 학과코드(HAKGWA_CD) 목록 / 쿠키 필수 여부 / TERM 코드 매핑 → 05_미해결_과제 P3.

[D-10] ★ 핵심 방향전환 검토 = 옵션 A (강의계획서 파싱 중심 재설계)
- 문제인식: 현재 시간표 구도는 크롤링+조합계산이 심장이고 Upstage가 곁가지 → 조원 드롭案과 같은 함정.
- 권장방향(옵션A): 무게중심을 강의계획서 PDF 파싱으로 이동.
  개설강좌 수집=최소 시딩, 진짜가치=강의계획서 Parse/Extract(평가방식·과제비중·시험·수업유형).
  → "강의계획서를 읽어 학기 부담까지 예측하는 시간표"로 재프레이밍.
- 대안: 옵션B(다른 후보 선회) / 옵션C(현구도+프레이밍 방어, 리스크 큼).
- 상태: ★ 팀 회의에서 결론낼 최우선 안건. 미확정. → [확정됨: D-12]

[D-12] 옵션 A 확정 + 정량지표 확정(평점 제외 v1) + 개발계획 착수 (2026-07-13)
- 결정: D-10의 옵션A(강의계획서 파싱 중심 재설계)를 최종 채택. 개설강좌 수집=최소 시딩,
  강의계획서 Document Parse/Extract가 서비스 심장.
- 정량지표(P0) 확정: "시간제약 + Upstage파생 부담도(과제비중·시험집중도·수업형태)" 2축으로 시작.
  기존 후보였던 "평점"은 v1 스코프에서 제외.
- 평점 데이터 소스(D-08/P1) 결정: 옵션C(v1은 평점 제외) 채택.
  - 이유: 데모데이(2026-07-25)까지 D-12로 시간이 빠듯함. 성대 성적분포 공시자료 확인(옵션A)이나
    사용자 직접입력 UI(옵션B)는 개발 리소스 대비 우선순위가 낮음. 평점 없이도 Upstage 파생
    부담도 지표만으로 "정량화된 추천" 차별화가 성립하므로 스코프 축소가 핵심가치를 해치지 않음.
  - 평점은 v2 확장 후보로 보존 (에타식 크롤링은 계속 금지, 04 참조).
- 후속: 개발 착수. 순서는 수집기(scraper) → Upstage 파싱 파이프라인 → 조합로직 → Solar 추천사유 → UI.
  Upstage 콘솔 실검증(P2)과 강의계획서 PDF 접근성 검증(P6)은 파싱 파이프라인 착수 시점에 최우선 처리.

[D-13] Phase 4 조합 로직 범위 축소 — 충돌·사용자 제약만 (2026-07-13)
- 결정: 이번 Phase 4에는 시간 충돌 제거와 사용자 제약(불가 요일, 최소 시작 시간)만 구현한다.
  Upstage 파생 부담도 점수화, 결과 정렬, "상위 N개 추천"은 구현하지 않는다.
- 이유: 시간표 유효성 규칙을 독립적으로 검증하고, 아직 확정되지 않은 부담도 가중치로 임의 순위를
  만들지 않기 위함. Document Parse 결과는 서비스 핵심으로 유지하되, 조합 추천 점수에 연결하는 일은
  후속 단계로 분리한다.
- 안전장치: 조합 수가 한도를 넘으면 일부 결과를 상위 N처럼 잘라 반환하지 않고, 사용자가 후보나
  제약을 더 좁히도록 명시적으로 실패시킨다.

[D-14] 사용자 중심 전체 플로우·누락 없는 책가방 모델 확정 (2026-07-13)
- 결정: 필수과목과 선택 책가방을 분리하고, 책가방마다 최소·최대 선택 수를 둔다. 과목은 학수번호
  단위이며 여러 분반을 소유하고, 선택된 과목에서는 정확히 한 분반만 고른다.
- 결정: 조합 생성은 결정론적 엔진이 담당하고 Solar는 유효 조합의 설명·정렬만 담당한다. AI가 책가방에서
  직접 조합을 생성하게 하지 않는다.
- 결정: 기수강 과목은 기본 제외하되 재수강 토글로 복원하고, 고정 일정은 이름 있는 시간 구간으로 둔다.
- 결정: 최종 제품에는 Solar 순위가 포함될 수 있으나 D-13의 현 구현 범위는 유지한다. 구체적인 순위
  기준과 가중치는 별도 검증 후 구현한다.
- 상세 구조·예외·캠퍼스 규칙은 `docs/06_제품_플로우_및_조합모델.md` 참조.

[D-15] 학사문서 AI 추출은 고정 JSON + 서버 검증 + 사용자 승인 (2026-07-13)
- 결정: Document Parse Markdown을 Solar에 보내되, JSON 전용 프롬프트로 필드를 제한하고
  서버가 AcademicProfile 타입을 런타임에 다시 검증·재구성한다.
- 안전장치: 이름·전체 학번·정확한 성적과 임의 출력 필드는 응답 객체에 복사하지 않는다.
  원본 파일과 전체 Parse Markdown은 API가 반환·영구저장하지 않는다.
- 적용 규칙: 자동 추출 결과는 항상 `draft`. 형식 오류 행은 묵인하지 않고 review issue로
  보존하며, 추천 로직은 후속 UI에서 사용자가 수정·확정한 결과만 사용한다.
- 구현: `POST /api/parse-academic-document` (`course_history` / `graduation_requirements`).

[D-16] 개설강좌는 파일 업로드 없이 사용자 선택 범위를 서버에서 즉시 조회 (2026-07-14)
- 결정: 개발용 `scraper JSON 불러오기`를 사용자 화면에서 제거하고, 기본정보의 학년도·학기·캠퍼스·
  학과코드를 `POST /api/skku-courses`에 전달한다.
- 이유: 사용자가 수집기나 JSON 파일을 알 필요가 없어야 하며, 브라우저가 성대 API 세션·SSV 형식에
  직접 의존해서도 안 된다. Next.js 서버가 세션을 성립하고 최소 Course 필드만 반환한다.
- 안전선: 전 학과를 미러링하지 않고 사용자가 확정한 학과 한 범위만 요청 시점에 조회하며 결과를
  저장하지 않는다. 성대 요청 사이에는 기존 수집기와 같은 간격을 둔다.
- 현재 범위: 전공 자동조회 완료. 교양 API는 `%` 캠퍼스 전체 조회가 0건이고 영역/과목코드가 필요하므로
  교양 영역 선택 UI와 함께 별도 연결한다.

[D-17] 교양은 공식 3단계 조회, 시간표 결과는 블록 주간표로 표시 (2026-07-14)
- 결정: 교양은 캠퍼스 전체를 임의 수집하지 않고 `영역 개설 수→영역 내 교과목→선택 교과목 분반`을
  사용자 선택 시점에만 조회한다. 전공 후보와 합친 뒤 같은 충돌·사용자 제약 엔진을 사용한다.
- 결정: 유효 조합은 텍스트 시간 문자열 대신 월~금 08:00~22:00 시각 블록으로 보여 준다.
  순위·점수·상위 N은 추가하지 않고 D-13 범위를 유지한다.
- 이유: 공식 정보광장과 같은 탐색 계층을 보존하면서 전과목 상시 미러링을 피하고, 사용자가 실제
  요일·시작·지속시간을 한눈에 확인할 수 있게 하기 위함.

[D-18] 공식 데모데이 루브릭을 제품·개발 우선순위 게이트로 채택 (2026-07-14)
- 결정: 운영진 공지 원문과 100점 평가표를 `docs/08_데모데이_평가항목_루브릭.md`에 보존하고,
  모든 AI 도구가 세션마다 필수로 읽는다.
- 결정: 큰 기능과 데이터 구조는 정보 출처·목적, 처리·검증 파이프라인, 범용 LLM 대비 가치,
