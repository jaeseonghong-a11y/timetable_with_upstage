■ ★★ 라이브 서버 검증 & 블로커 해결 (2026-07-12, 브라우저 fetch로 실증)
개발 중 발견된 "ErrorCode:int=0(성공)인데 과목 데이터 0건" 블로커의 원인 3가지를 실측으로 규명함.
경영학과(316901)·2학기(TERM=20) 기준 정상 응답은 214행. 아래 조건이 어긋나면 0건이 된다.

★ 원인 1 (핵심): ROAD_MAP 값 "%"의 URL 인코딩 깨짐
  - ROAD_MAP=%   → 정상 (214행). %는 SSV에서 전체(와일드카드) 의미.
  - ROAD_MAP=%25 → 0행!! (퍼센트가 URL 인코딩되면 조건이 아무것도 매칭 안 됨)
  - ROAD_MAP=""  → 0행
  - ROAD_MAP 자체를 빼면 → ErrorCode:-904 (에러)
  ⇒ Python requests가 바디를 URL 인코딩하면서 % → %25 로 바꾸면 이 증상이 난다.
  ⇒ 해결: 바디를 문자열로 넘겨 requests가 재인코딩하게 두지 말고, **UTF-8 raw bytes로 인코딩해서
     data=body.encode("utf-8") 로 전송**한다. (client.py는 이미 .encode("utf-8") 사용 — 이 방식이 맞음)
     requests에서 data=dict 형태로 주면 폼 인코딩되어 %가 깨지므로 절대 금지. 반드시 raw string/bytes.

★ 원인 2: 학과-캠퍼스(CAMPUS_GB) 짝이 맞아야 함
  - 건축학과(316307) + CAMPUS_GB=2(자연과학) → 112행 ✅

  ⇒ 학과마다 소속 캠퍼스가 정해져 있다. 학과코드에 맞는 캠퍼스로 조회해야 한다.

⇒ 결론: 블로커 원인은 "요청 바디 인코딩"이 맞았고, 구체적으로 % 문자 URL 인코딩이 범인.

■ ★★★ P3-a 블로커 진짜 원인 = 세션 (2026-07-12 재검증, 최종 규명)
위 "% 인코딩"도 실재하는 함정이지만, "client.py는 이미 raw bytes인데도 0행" 이라는 재현실패의
진짜 원인은 별도였다. 브라우저 성능로그(performance.getEntriesByType)로 실제 호출 순서를 확인한 결과:

  실제 페이지가 selectMain.do 전에 부르는 요청 순서:
    1. /gaia/E_NCommon/sessionLogin.do    ← ★ 세션 성립 (JSESSIONID 발급)
    2. /gaia/E_NCommon/refreshSession.do  ← 세션 갱신
    3. /gaia/E_NHSSU030840M/selectMenutree.do
    4. /gaia/E_NHSSU900020M/selectMain.do  ← 세션이 있어야 데이터 반환

★ 원인: selectMain.do는 "유효한 세션"이 있어야 데이터를 준다.
  - 세션 없이 호출 → ErrorCode:int=0(성공) BUT 데이터 0행, 응답 806바이트(컬럼 스키마만).
  - 브라우저 fetch가 214행을 준 건, 페이지가 이미 sessionLogin.do를 불러 세션이 성립돼 있었기 때문.
  - curl/Python이 0행인 건 sessionLogin.do를 안 불러 세션이 없어서. (페이지 GET만으론 세션 안 생김)

★ sessionLogin.do 확인:
  - POST /gaia/E_NCommon/sessionLogin.do (빈 SSV 바디 "SSV:utf-8"+RS 로도 OK)
  - 응답 200, 약 389KB, ErrorCode:0, Dataset:gdsUser(사용자정보)·gdsMsg(메시지) 등 반환.
  - 이 호출이 JSESSIONID(httponly) 세션 쿠키를 발급/유효화한다.
    (참고: WMONID는 JS에서 보이지만 JSESSIONID는 httponly라 document.cookie엔 안 보임)

★★ 해결책 (Python 수집기):
  requests.Session() 을 써서 쿠키를 자동 유지하고, 아래 순서로 호출한다:
    session = requests.Session()
    session.headers.update({브라우저 UA, Referer, Origin, ...})
    session.post(BASE + "/gaia/E_NCommon/sessionLogin.do", data=("SSV:utf-8"+RS).encode())  # 세션 성립
    # (선택) refreshSession.do 도 한 번
    resp = session.post(BASE + ENDPOINT_MAJOR, data=body.encode("utf-8"))  # 이제 214행 나옴
  → 즉 client.py에 "세션 초기화(ensure_session)" 단계를 추가해야 한다.
    한 번 sessionLogin 하면 그 Session 객체로 이후 여러 학과를 연속 조회 가능.

★ 요약 (0행이 나올 때 점검 순서):
  1. sessionLogin.do 를 먼저 호출했는가 (★ 가장 흔한 원인)
  2. requests.Session()으로 쿠키를 유지하는가
  3. ROAD_MAP=% 가 %25로 안 깨졌는가 (raw bytes 전송)
  4. 학과-캠퍼스(CAMPUS_GB) 짝이 맞는가
  5. 학과코드·연도·학기가 유효한가

■ ★★★★ 헤더 정밀 진단 (2026-07-12, 브라우저 변형 실험)
"세션까지 완벽 재현했는데도 0행"에 대해, 브라우저(세션 성립됨)에서 헤더/바디를 하나씩 바꿔가며
무엇이 필수이고 무엇이 무관한지 실측함. 214행 성공 요청의 완전한 바디(경영학과·TERM=20):

  성공 바디 (171 bytes, 구분자 ␞=0x1E):
    SSV:utf-8␞YEAR=2026␞TERM=20␞HAKGWA_CD=316901␞ROAD_MAP=%␞CAMPUS_GB=1␞HAK_JIBJUNG=0␞
    _FIRST_OUT_DS_NM=dsGrdMain␞_TRANSACTION_ID=selectMain␞_ALL_OUT_DS_NM=dsGrdMain=dsGrdMain␞

  ★ 필수로 확정된 것 (없으면 0행):
    - X-NX-Content-Type: 2   ← ★★ 이게 유일한 필수 헤더. 값이 반드시 "2".
      · 값 "1" 또는 "0" → 0행 / 헤더 자체 없음 → 0행 / 대소문자(x-nx-content-type)는 무관
      · 이 헤더가 서버에 "바디는 SSV 포맷"임을 알려 파싱 방식을 결정하는 것으로 보임.
    - ROAD_MAP=% (와일드카드, %25로 깨지면 0행)

  ★ 무관한 것 (바꿔도 214행 — 즉 이게 원인이 아님):
    - Content-Type: text/xml / text/xml;charset=UTF-8 / application/octet-stream /
      application/x-www-form-urlencoded → 전부 214행 (Content-Type은 아무거나 OK)
    - 바디 첫 줄: SSV:utf-8 / SSV:UTF-8 / SSV → 전부 214행 (대소문자·charset명 무관)
    - X-Requested-With: 있으나 없으나 214행 (필수 아님)

  ⇒ 결론: 브라우저(세션 있음)에선 X-NX-Content-Type:2 하나만 맞으면 나머지는 매우 관대하다.
     따라서 Claude Code가 세션까지 맞췄는데 0행이라면, 남은 유력 원인 3가지:
       (1) X-NX-Content-Type: 2 헤더가 실제 전송에 안 실렸거나 값이 다름
           → requests에서 headers dict에 정확히 "X-NX-Content-Type":"2" 넣었는지,
             Session.headers 병합 과정에서 덮어써지지 않았는지 확인. 실제 전송 헤더를 로깅해볼 것.
       (2) selectMenutree.do 가 세션에 "메뉴 컨텍스트(PGM_ID/MENU_ID)"를 심는데 그게 selectMain 전에
           필요할 수 있음 → 바디의 _MENU_ID/_PGM_ID 를 실제값(NHSSU030840M 등)으로 채워볼 것.
           (브라우저 성공 요청 바디엔 _MENU_ID/_PGM_ID 가 비어 있었으나, 세션에 이미 심겨서 생략 가능했을 수)
       (3) TLS/HTTP2 지문 차이(가능성 낮음): 위 실험상 바디·헤더가 이렇게 관대한데 지문만으로
           거르는 정황은 약함. (1),(2)를 먼저 배제할 것.
  ⇒ 다음 액션: Claude Code가 requests로 보낼 때 "실제 전송된 요청 헤더 전체"를 덤프해서
     X-NX-Content-Type:2 가 진짜 실렸는지부터 확인. 이게 제일 유력.

8. GLS 학사문서 Document Parse→Solar 통합 검증 (2026-07-13)
═══════════════════════════════════════

- 공식 Console 예제의 `POST https://api.upstage.ai/v1/chat/completions`, `model=solar-pro3`,
  Bearer 인증 형식을 사용했다.
- `web/.env.local`의 키를 출력하지 않고 가상 빈 학사 데이터로 Solar를 실호출한 결과:
  HTTP 200, 응답 모델 `solar-pro3-260323`, 고정 JSON 3개 배열 파싱 성공.
- 공식 예제에서 JSON Schema `response_format` 지원은 확인하지 못했으므로 기능을
  가정하지 않았다. JSON-only 프롬프트 뒤 서버 런타임 검증으로 잘못된 출력을 차단한다.

  Document Parse→Solar를 서버에서만 호출하고 `status=draft` AcademicProfile만 반환한다.
- 원본·전체 Parse Markdown은 응답·로그·파일에 남기지 않는다. Solar가 형식 밖 행을
  내면 해당 행을 반영하지 않고 `reviewIssues`로 넘긴다.
