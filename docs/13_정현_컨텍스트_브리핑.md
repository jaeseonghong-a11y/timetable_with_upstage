[D] 왜 이게 어려운 문제인지 (핵심 — 사용자가 강조한 부분)
────────────────────────────────────────
이 작업의 본질적인 어려움은 "AI로 PDF를 읽는 것" 자체가 아니라,
**서로 완전히 다른 구조를 가진 데이터를 하나의 일관된 체계로 정리하는
것**입니다.

1. **로드맵 PDF 자체가 학과마다 포맷이 다릅니다.**
   - 나노공학과: 비교적 단순한 1단계 카테고리 구조 (테스트 완료)
   - 건축학과: 대분류→소분류 2단계 구조, 학기 구분이 안 보이는 레이아웃,
     5년제 과정이라 학년 범위 자체가 다름 (미검증 — 다음 우선순위)
   - 의예과/의학과: 아직 손도 안 댐. 의학 계열은 학제가 또 다를 가능성이
     높음
2. **복수전공/연계전공은 로드맵 PDF와 아예 다른 데이터 소스·구조를
   씁니다.** ([B]-③) `curriculum.py`로 가져오는 데이터는 "학년/학기별
   배치"가 없는 **평평한(flat) 과목 목록**입니다. 로드맵 PDF는 "학년별
   격자(grid)"입니다. 이 둘을 억지로 같은 표에 우겨넣으면 정보 손실이
   생깁니다 — grid형과 flat형을 **서로 다른 구조로 인정하고, 학과코드로
   연결**하는 설계가 필요합니다.
3. **연계전공은 여러 학과가 결합된 개념이라 "소속 학과" 자체가 모호할 수
   있습니다.** 확인된 연계전공은 13개입니다(`docs/02` 6절). 이것들이
   로드맵 PDF에도 별도 페이지로 존재하는지, 아니면 아예 없는지부터 확인이
   필요합니다.
4. **최종적으로 이 데이터가 기존 시스템과 "조합"되어야 합니다** — 예를
   들어 나중에 "로드맵상 이번 학기에 들어야 할 과목을 AI 추천에서
   우선한다"는 기능을 만들려면, 이 로드맵 데이터가 기존 `Requirement`/
   과목 카탈로그와 학과코드·과목명(또는 학수번호) 기준으로 매칭되어야
   합니다. 지금은 과목명 표기가 로드맵 PDF와 실제 개설강좌 시스템에서
   다를 수 있다는 점도 감안해야 합니다(예: 약어, 띄어쓰기 차이).

[E] 지켜야 하는 제약 (절대 원칙)
────────────────────────────────────────
- **원본 PDF는 절대 커밋하지 않습니다.** 학교 발행 저작물일 가능성이
  있어서, 로컬(`private/` 또는 `.gitignore` 대상)에만 둡니다. 반면
  **정규화 결과(JSON)는 공개 자료 재가공물이라 커밋 가능**합니다 —
  개인 학사문서(사용자 업로드분)와는 다른 취급입니다.
- **현재 학년을 과목 코드나 설명으로 임의 추정하지 않습니다.** 로드맵에
  학년 정보가 없는 항목은 `null`로 두고 지어내지 않습니다.
- **로드맵 데이터는 "확정 소스"가 아니라 "참고 자료"입니다.** 실제
  졸업요건 충족 여부는 어디까지나 사용자가 업로드한 GLS 문서(졸업요건
  충족현황) Parse 결과가 우선합니다 (`docs/02` 6절 "사용 원칙"에 이미
  명시된 원칙). 로드맵은 "다음에 뭘 들으면 좋을지 참고"용이지, 졸업
  가능 여부를 이 데이터로 판정하면 안 됩니다.
- 불확실한 항목은 조용히 넘기지 말고 `uncertain`/`review` 같은 플래그로
  명시적으로 표시합니다 (기존 GLS 문서 처리 관례와 동일).
- API 키·서버 로직 구현은 정현님 범위가 아닙니다. 실제 코드는 재성이
  Claude Code로 짜므로, 정현님은 **스키마 설계 + 검증 + (필요하면)
  수작업 프롬프트/파이프라인 설계**까지가 산출물입니다.

[F] 이번 작업의 목표 (사용자가 정의한 3가지 축)
────────────────────────────────────────
1. **정확하게 정리** — 서로 다른 학과·형식의 로드맵 데이터를 신뢰할 수
   있는 정규화된 구조로 만든다 (환각·누락·오분류 최소화, 불확실하면
   플래그).
2. **어떻게 조합** — grid형(로드맵 PDF, 학년·학기 있음) + flat형
   (복수전공/연계전공 과목 목록, 학년·학기 없음) + 기존 사용자 졸업요건
   데이터, 이 세 가지를 학과코드·과목명(or 학수번호) 기준으로 서로
   연결 가능한 구조로 설계한다.
3. **어떻게 활용** — 이 데이터가 실제로 서비스의 어느 기능에 쓰일지
   설계한다. 이미 사용자가 승인한 활용처 후보 4가지 (아직 미구현,
   백로그): 학년별 로드맵 우선 표시 / AI 추천 시 로드맵 기반 과목
   보충 / 이수 진도율 현황판 / Solar 추천 이유에 로드맵 근거 추가.
