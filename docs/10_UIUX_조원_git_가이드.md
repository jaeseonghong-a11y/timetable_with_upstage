■ UI/UX 작업자를 위한 Git/GitHub 완전 초보 가이드
════════════════════════════════════════════════════════════════
대상: Git·GitHub·"바이브코딩"을 처음 써보는 UI/UX 담당 조원
목적: 기존 서비스 코드를 망가뜨릴 걱정 없이, 안전하게 UI/UX를 실험하고
      완성되면 재성(팀 코딩 담당)에게 넘기는 방법
최종 갱신: 2026-07-17

이 문서만 따라 하면 됩니다. 중간에 막히면 5번(막혔을 때)으로 바로 가세요.


────────────────────────────────────────
0. 왜 이렇게까지 해야 하나요? (딱 1분만 읽기)
────────────────────────────────────────
지금 서비스는 실제로 배포되어 있고(https://timetable-with-upstage.vercel.app/),
재성이 계속 다른 기능을 고치고 있습니다. 만약 규동님이 그 코드를 직접 열어서
바로 고치면:
  · 재성이 작업 중이던 부분과 파일이 겹쳐서 서로 방해가 될 수 있고
  · 실수로 뭔가 지워도 원래대로 되돌리기 어렵고
  · 지금 실제로 열려 있는(배포된) 서비스가 그대로 망가질 수 있습니다

그래서 "브랜치(branch)"라는 격리된 작업 공간을 하나 만들어서, 거기서만
마음껏 실험하고, 다 되면 재성에게 "이거 확인해줘"라고 넘기는 방식을 씁니다.
이 방식을 쓰면 **실수해도 원본은 절대 망가지지 않습니다.** 마음 편하게
실험하셔도 됩니다.


────────────────────────────────────────
1. 딱 4개만 알면 됩니다 (개념)
────────────────────────────────────────
① 브랜치(branch)
   = 원본을 복사한 "나만의 작업 공간". 여기서 뭘 바꿔도 원본(main)은
     전혀 안 바뀝니다. 마치 문서 파일을 "사본으로 저장"해서 사본만
     고치는 것과 비슷합니다.

② 커밋(commit)
   = "여기까지 저장" 버튼. 작업하다가 어느 정도 됐다 싶으면 커밋을
     합니다. 커밋을 자주 할수록, 나중에 "이 시점으로 돌아가기"가 쉬워집니다.

③ 푸시(push)
   = 내 컴퓨터에만 있는 커밋을 GitHub(인터넷 서버)에 올리는 것.
     푸시를 해야 재성이나 Vercel(배포 서비스)이 내 작업을 볼 수 있습니다.

④ PR(Pull Request, "풀리퀘스트")
   = "제 작업 다 됐어요, 원본(main)에 합쳐주세요"라고 요청하는 것.
     GitHub 웹사이트에서 버튼 몇 번 클릭하면 됩니다. PR을 올린다고
     바로 합쳐지는 게 아니라, 재성이 확인하고 승인해야 합쳐집니다.

정리하면 흐름은: 브랜치 만들기 → (그 안에서) 작업 → 커밋 → 푸시
→ Vercel 프리뷰로 눈으로 확인 → 다 되면 PR


────────────────────────────────────────
2. 처음 딱 한 번만 하는 준비
────────────────────────────────────────

### 2-1. 재성이 먼저 해줘야 하는 것 (규동님은 기다리기만 하면 됨)
  □ GitHub 저장소에 "협업자(Collaborator)"로 초대받기
    → 재성이 GitHub 저장소 Settings → Collaborators에서 규동님 GitHub
      계정을 초대합니다. 이메일로 초대장이 오면 수락하면 끝.
  □ (없다면) GitHub 계정 만들기: https://github.com 에서 무료 가입

### 2-2. 내 컴퓨터에 설치해야 하는 것
  □ Git 설치: https://git-scm.com/downloads (Windows면 "Git for Windows")
  □ Node.js 설치: https://nodejs.org (LTS 버전 다운로드) — 서비스를
    내 컴퓨터에서 직접 실행해서 눈으로 확인하려면 필요합니다.
  □ (추천) Visual Studio Code: https://code.visualstudio.com — 코드/CSS를
    눈으로 편하게 보고 고칠 수 있는 에디터

  ※ 터미널(명령어 입력 창)이 낯설다면, Git 명령어 대신 **GitHub Desktop**
    (https://desktop.github.com) 이라는 프로그램을 써도 됩니다. 버튼
    클릭만으로 브랜치 만들기/커밋/푸시가 가능합니다. 아래 명령어들은
    전부 GitHub Desktop 안의 버튼으로도 똑같이 할 수 있습니다.
    (막히면 CLI 대신 GitHub Desktop 쓰겠다고 재성에게 말하세요.)

### 2-3. 저장소 내려받기 (딱 한 번만)
터미널을 열고 원하는 폴더로 이동한 뒤:
```bash
git clone https://github.com/jaeseonghong-a11y/timetable_with_upstage.git
cd timetable_with_upstage/web
npm install
```
`npm install`은 시간이 좀 걸립니다(1~5분). 이건 딱 한 번만 하면 됩니다.


────────────────────────────────────────
3. 작업할 때마다 반복하는 순서
────────────────────────────────────────

### 3-1. 내 작업용 브랜치 만들기 (작업 시작할 때 한 번)
```bash
git checkout main          # 먼저 원본 기준으로 최신화
git pull origin main       # 재성이 그동안 고친 것도 받아오기
git checkout -b uiux-redesign   # "uiux-redesign"이라는 내 브랜치 생성 + 이동
```
브랜치 이름은 원하는 대로 지어도 되지만, `uiux-` 로 시작하면 나중에
알아보기 쉽습니다. (예: `uiux-timetable-card`, `uiux-color`)

**한 번 브랜치를 만든 뒤에는, 다음 날 다시 작업할 때 `checkout -b`를
또 안 해도 됩니다.** `git checkout uiux-redesign` (-b 없이)으로 그
브랜치로 돌아가기만 하면 됩니다.

### 3-2. 로컬에서 서비스 띄워서 보면서 작업하기
```bash
npm run dev
```
터미널에 `http://localhost:3000` 같은 주소가 뜹니다. 브라우저에서 그
주소를 열면, **내 컴퓨터 안에서만** 지금 코드로 서비스가 실제로 돌아갑니다.
파일을 저장할 때마다 브라우저가 자동으로 새로고침되면서 바뀐 내용이
바로 보입니다. (인터넷에 올라가는 게 아니라 내 화면에서만 보이는 것이라
아무리 이상하게 고쳐도 안전합니다.)

### 3-3. 어떤 파일을 고치면 되는지
| 하고 싶은 것 | 건드릴 파일 | 충돌 위험 |
|---|---|---|
| 색, 여백, 폰트, 버튼 크기·배치, 애니메이션 | `web/src/components/TimetablePlanner.module.css` | 거의 없음 (추천) |
| 카드 순서/구조를 크게 바꾸기, 새 화면 흐름 | `web/src/components/TimetablePlanner.tsx` | 있음 (아래 참고) |

  · `.module.css` 파일은 스타일만 모여 있는 파일이라, 여기만 고치면
    재성이 로직을 고치는 부분과 거의 안 겹칩니다. **가능하면 이 파일
    위주로 작업해주세요.**
  · `.tsx` 파일(실제 화면 구조/버튼 배치 등)은 재성의 기능 코드와 같은
    파일에 섞여 있어서, 여기를 많이 고치면 나중에 합칠 때 "충돌"이 날
    수 있습니다. 구조를 크게 바꾸고 싶다면, 미리 재성에게 "이 부분을
    별도 파일로 빼줄 수 있어?"라고 요청하세요. (가능한 작업입니다.)
  · 잘 모르겠으면 그냥 고치고 재성에게 물어봐도 됩니다 — 충돌이 나도
    사라지는 건 없습니다 (4번 참고).

### 3-4. 어느 정도 됐으면 저장(커밋)하고 GitHub에 올리기(푸시)
```bash
git add .
git commit -m "카드 색상과 여백 조정"
git push origin uiux-redesign
```
- `git add .` : 바뀐 파일 전부를 "저장할 목록"에 추가
- `git commit -m "..."` : 그 시점 상태를 저장. 따옴표 안에는 뭘 바꿨는지
  간단히 한글로 적으면 됩니다.
- `git push origin uiux-redesign` : 내 브랜치를 GitHub에 업로드

**하루에 여러 번 해도 됩니다.** 오히려 자주 커밋+푸시하는 게 좋습니다
(작업이 조금씩 저장되니 안전합니다).

### 3-5. Vercel 프리뷰 링크로 실제로 확인하기
푸시하면(브랜치가 처음 올라가면) GitHub 저장소에서 Vercel이 자동으로
그 브랜치 전용 미리보기 웹사이트를 만들어 줍니다(재성이 미리 설정해
두면). 이 링크는:
  · 실제 서비스(timetable-with-upstage.vercel.app)와 완전히 분리된
    "미리보기용" 주소입니다.
  · 원본 서비스는 전혀 영향받지 않습니다.
  · 이 링크를 재성이나 다른 조원에게 공유해서 "이렇게 바꿨는데 어때?"
    라고 보여줄 수 있습니다.

이 링크를 못 찾겠으면 재성에게 물어보세요 — 처음에 Vercel 쪽 설정을
한 번 확인해줘야 할 수도 있습니다.

### 3-6. 다 완성되면: Pull Request(PR) 만들기
1. GitHub 저장소 페이지(https://github.com/jaeseonghong-a11y/timetable_with_upstage)
   접속
2. 상단에 "Compare & pull request" 라는 노란 버튼이 보이면 클릭
   (안 보이면 "Pull requests" 탭 → "New pull request" → base: main,
   compare: uiux-redesign 선택)
3. 제목/설명에 뭘 바꿨는지 간단히 적기 (예: "타임테이블 카드 색상·간격 개선")
4. "Create pull request" 클릭
5. 재성에게 "PR 올렸어요"라고 알려주기 → 재성이 확인 후 main에 합침

**여기까지만 하면 규동님 역할은 끝입니다.** 실제로 main에 합치는 건
재성이 확인하고 진행합니다.


────────────────────────────────────────
4. 걱정되는 것들 미리 답변
────────────────────────────────────────

Q. 제가 고친 게 재성님이 고친 걸 지워버리진 않나요?
A. 아니요. GitHub은 "파일을 통째로 바꿔치기"하는 게 아니라, 서로 다른
   부분을 자동으로 합쳐줍니다. 정확히 같은 줄을 둘 다 고쳤을 때만
   "충돌"이 뜨는데, 이때도 조용히 한쪽이 사라지는 게 아니라 사람이
   확인할 때까지 멈춰 있습니다. 그리고 이 충돌 해결은 재성이 처리하니
   규동님이 신경 쓸 필요 없습니다.

Q. 실수로 뭔가 잘못 고치거나 지웠어요. 되돌릴 수 있나요?
A. 네, 여러 단계로 안전장치가 있습니다.
   · 아직 커밋 전이면: 그냥 파일을 원래대로 고치거나, 저장 안 하면 됩니다.
   · 이미 커밋했지만 아직 재성에게 안 넘겼으면: 신경 쓰지 말고 계속
     작업하다가 다시 고치면 됩니다. 어차피 PR을 올리기 전이라 원본에는
     영향 없습니다.
   · PR까지 올렸는데 이상한 것 같으면: 재성에게 "이 PR 이상한 것 같아,
     다시 볼게"라고 말하면 됩니다. 병합(merge) 전이면 언제든 취소하거나
     고칠 수 있습니다.
   · 즉, **main(원본)에 실제로 합쳐지기 전까지는 아무리 실수해도 100%
     안전**합니다. 심지어 합쳐진 후에도 재성이 원클릭으로 되돌릴 수
     있습니다.

Q. 브랜치를 잘못 만들었거나 꼬인 것 같아요.
A. 당황하지 말고 재성에게 지금 화면을 캡처해서 보내주세요. 대부분
   몇 분 안에 해결 가능한 문제입니다. 혼자 `reset`, `force`, `delete`
   같은 명령어를 검색해서 따라 하지 마세요 (5번 참고).


────────────────────────────────────────
5. 절대 하면 안 되는 것 (이것만 지키면 안전합니다)
────────────────────────────────────────
아래 명령어들은 검색하면 나올 수 있지만, **절대 스스로 실행하지 마세요.**
잘못 쓰면 작업 내용이 진짜로 사라질 수 있습니다. 필요하다면 반드시
재성에게 먼저 물어보세요.

  ✗ git push --force / git push -f
  ✗ git reset --hard
  ✗ git checkout main -- . (다른 브랜치 내용으로 파일 통째로 덮어쓰기)
  ✗ git clean -f
  ✗ main 브랜치에서 직접 작업하고 바로 push (항상 내 브랜치에서만 작업)
  ✗ 파일을 통째로 복사해서 다른 사람 폴더에 덮어씌우기

**위 5개만 안 하면, 3번 순서(브랜치 → 작업 → 커밋 → 푸시 → PR)를 아무리
반복해도 절대 원본이 망가지지 않습니다.**


────────────────────────────────────────
6. 막혔을 때
────────────────────────────────────────
터미널에 빨간 글씨(에러)가 뜨거나, 뭘 해야 할지 모르겠으면:
  1. 당황하지 말고 지금까지 한 작업(파일)은 그대로 두세요 (지우거나
     되돌리려고 하지 마세요).
  2. 에러 메시지 전체를 스크린샷으로 찍기
  3. 재성에게 "여기서 막혔어요" + 스크린샷 + 지금까지 뭘 했는지 한 줄로
     보내기
  4. 재성이 원격으로 확인하고 해결법을 알려줄 것입니다.

**모르는 걸 혼자 해결하려고 인터넷에서 명령어를 따라 치는 것보다,
막힌 그 상태로 두고 물어보는 게 훨씬 안전합니다.**
