■ 사례 1. Solar 졸업요건충족현황 추출이 매번 다르게 나오는 문제
═══════════════════════════════════════

**문제상황**
중간점검 멘토링에서 지적받은 문제(Q2): 같은 졸업요건충족현황 문서를 올려도 "검토 필요" 항목이
어떨 때는 0개, 어떨 때는 엄청 많이 나오고, 내용도 매번 다르고, 오류가 아닌데 오류라고 판단하는
경우가 있었다.

**시행착오·실측 과정**
1. 코드를 먼저 정독해서 구조 파악: 이미 "결정론적 표 파서(코드) + Solar(AI)" 하이브리드
   구조였음을 확인. 즉 AI에게 통째로 맡기는 구조가 아니었다.
2. 사용자가 실제 본인의 졸업요건충족현황 캡처를 제공 → 그 표를 그대로 재구성해서 **실제
   Upstage API에 동일한 입력을 3번 반복 호출**하는 재현 스크립트를 작성해 실측.
3. 실측 결과, 놀라운 사실 발견: **표를 정확히 읽었을 때조차** Solar가 붙이는 "검토 필요 이유"가
   매번 달랐다. 특히 한 번은 `["string"]`이라는 **JSON 스키마 설명 문구를 그대로 베낀** 값을,
   또 한 번은 프롬프트 지시문 자체를 그대로 되풀이하는 값을 내놓았다 — 이건 문서 내용과
   무관한, AI의 순수한 "형식 오작동"이었다.
4. 해결 방향을 웹서칭으로 조사: Upstage Solar(solar-pro3)가 OpenAI 스타일의
   "구조화 출력(JSON Schema strict mode)"을 지원하는지 검색 결과가 엇갈려서, **직접 실제
   API를 호출해 검증**. 지원 확인.
5. 구조화 출력을 적용해 재실측 → **형식 오작동(스키마 문구 베끼기)은 100% 사라짐**. 하지만
   "몇 개 항목을 반환할지·충족 여부를 어떻게 판단할지"의 비일관성은 여전히 남아있음을 확인
   (같은 입력인데 항목을 통째로 빠뜨리거나, 충족 상태가 실행마다 달라짐).
6. 최종 처방: 표 파서가 이미 정확히 처리한 항목에 대해서는 **Solar의 의견(검토 이유·이슈)을
   아예 반영하지 않고, 코드가 계산한 값만 신뢰**하도록 구조를 바꿈. (Solar는 그래도 "이 시간표
   추천이 왜 좋은지" 같은 설명 역할에는 계속 쓰임 — 여기서만 "판정" 역할을 뺀 것.)

**효과 (수정 전 / 후 실측 비교, 동일 문서로 API 3회 반복 호출)**

| 항목 | 수정 전 | 수정 후 |
|---|---|---|
| 검토 필요 이유(reviewReasons) | 3회 실행 결과가 전부 다름 (없음 / 프롬프트 문구 echo / `["string"]` 15개) | **3회 실행 결과가 바이트 단위로 완전히 동일** |
| 문서 단위 이슈 | 3회차에 존재하지도 않는 변수명을 지어내는 현상 발견 | 동일하게 제거 |
| 충족 상태(충족/미충족 등) | 대체로 안정적이었으나 항목별 편차 존재 | 15개 항목, 3회 모두 완전히 동일 |

→ **"AI가 매번 다르게 판단하는 것 같다"는 사용자 체감을, 실제 API를 반복 호출하는 실측으로
재현·정량화하고, 처방 후 다시 실측으로 재현 안 됨을 확인**한 것이 이 사례의 핵심.

■ 사례 2. 같은 증상, 다른 원인 — 수강/취득과목에서도 재현했더니
═══════════════════════════════════════

**문제상황**
사용자가 "졸업요건충족현황과 같은 문제가 수강/취득과목에도 있다"고 지적. "같은 해결법을 써야
한다는 뜻은 아니다"라며, 동일한 실측 방법론으로 원인을 다시 조사하도록 요청.

**시행착오·실측 과정**
1. 코드부터 재확인: 수강/취득과목 쪽 병합 로직(`supplementCompletedCoursesFromTable`)이
   졸업요건 쪽과 **정반대 구조**였다 — 졸업요건은 "표가 주인공", 수강/취득과목은 "Solar가
   주인공, 표는 보조"로 짜여 있었음. 같은 증상이라도 원인 위치가 다를 수 있다고 판단.
2. 사용자가 실제 본인 수강/취득과목 PDF(개인정보 포함)를 제공 → **이름·학번 등 민감정보는
   제거**하고 과목 데이터(33과목, 학점 합계 94.0)만 원문과 정확히 일치하게 재구성해 동일한
   실측(API 3회 반복) 진행.
3. 1차로 졸업요건과 같은 방식(표 파서 우선)으로 수정했는데 **재실측해도 효과가 전혀 없었다.**
4. 원인을 추가로 추적한 결과, 더 근본적인 버그 발견: **수강/취득과목의 표 파서가 애초에
   HTML 형식 표를 읽지 못하고 있었다.** 졸업요건 표 파서는 마크다운 표와 HTML 표(`<table>`)
   둘 다 처리하도록 짜여 있었지만, 이쪽은 마크다운 표만 처리하도록 짜여 있어서 Document
   Parse가 HTML로 표를 뽑아주면 **표 파서가 0행을 인식하고 조용히 포기** — 그 결과 처방을
   적용할 데이터 자체가 없었던 것.
5. HTML 표 파싱 지원을 추가한 뒤에야 원래 처방(표 우선 반영)이 실제로 작동하기 시작.

**효과 (수정 전 / 후 실측 비교, 실제 문서 33과목으로 API 3회 반복 호출)**

| 항목 | 수정 전 | 수정 후 |
|---|---|---|
| 검토 필요 이유 | 빈 칸을 이유로 지어내거나 → 성적을 이유라고 반복 → 프롬프트 문구를 그대로 echo | **33과목 전부, 3회 모두 완전히 빈 배열** |
| 이수 상태 | 0학점 P등급(패스) 과목 하나가 "이수완료"↔"검토필요"를 오갔음 | 33과목 전부, 3회 모두 "이수완료"로 동일 |

→ **"같은 증상이니 같은 해결법이 통하겠지"라는 가정을 실측으로 검증했고, 실제로는 틀렸다는 걸
확인한 뒤 진짜 원인을 다시 추적한 사례.** 표면 증상만 보고 처방하지 않고, 매 단계 실제 API
호출로 확인해가며 고쳤다는 게 핵심 포인트.


═══════════════════════════════════════
■ 사례 3. 학과 목록이 실제와 다른 문제
═══════════════════════════════════════

**문제상황**
사용자가 "개설강좌 사이트와 우리 사이트의 소속학과 목록이 다르다, 누락이 있는 것 같다"고 지적.

**시행착오·실측 과정**
1. 코드 확인: 우리 사이트의 학과 목록이 **정적 스냅샷 파일(110개)** 이었다.
2. 이미 만들어져 있던 성균관대 실시간 학과 조회 스크레이퍼로 **API를 직접 재호출**해서
   실제 최신 목록을 뽑아봄 → **130개**. 22개가 통째로 빠져 있었다.
3. 대조해보니 동아시아학술원 소속 전공 전체와, 최근 신설된 융합/연계전공 다수가 누락돼
   있었다 — 정적 스냅샷이 갱신된 이후 학과가 신설·개편된 것.
4. 목록만 채워 넣고 끝내지 않고, 추가 문제를 발견: 새로 채워진 학과 중 다수가 4자리·8자리
   학과코드였는데, 검증 로직 3곳이 전부 "6자리 고정"으로 하드코딩돼 있어서 **목록에 넣어도
   실제 선택하면 막혔을 것**임을 코드 검토로 미리 발견하고 같이 수정.

**효과**
학과 드롭다운 130개로 정확화(22개 복구, 그중 동아시아학술원은 전체가 누락돼 있었음). 4/8자리
학과코드 선택 시 조회가 막히던 잠재적 버그도 실제로 발생하기 전에 함께 제거.

■ 이 세 사례가 보여주는 것 (발표 포인트)
═══════════════════════════════════════

- **"AI가 이상하다"는 막연한 사용자 체감을 그냥 넘기지 않고, 실제 API를 반복 호출해 재현·
  정량화한 뒤에 처방했다.** (감으로 고치지 않음)
- **같은 증상이라도 원인이 다를 수 있다는 걸 실측으로 직접 확인**했다(사례 2) — 첫 번째
  처방이 실측 재검증에서 효과가 없다는 게 드러나자, 가정을 버리고 원인을 다시 추적함.
- 세 사례 모두 **AI(Solar)에게 판단을 맡기는 범위를 줄이고, 코드가 확실히 계산할 수 있는
  부분은 코드가 담당**하도록 옮기는 방향으로 수렴했다 — 이는 중간점검 멘토링에서 받은 조언
  ("계산은 코드로, AI는 설명·해석만")과 정확히 일치하는 결론을 독립적으로 다시 확인한 것.
- 수정 전/후를 **같은 조건에서 여러 번 반복 호출해 비교하는 방식**으로 "효과가 있었다"를
  주장이 아니라 수치로 증명했다.
