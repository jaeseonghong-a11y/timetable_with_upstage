  · 기업 레벨로 가면 정보가 민감해서 **사내에 서버를 구축**해 쓰는 경우가 많다.

  실제로 저장하지 않는 구조여야 하므로 ①은 사실상 우리가 스스로 보장해야 하는 항목.

      [녹취: "스마트 쿠폰" → 스타벅스 쿠폰 등으로 추정]
  · 현업(데이터 애널리스트·사이언티스트)의 실제 활용 = **사용자 퍼널 분석**

    - "분석에 시간이 오래 걸린다 하면 **그동안 다른 걸 할 걸 쥐어주는**" 판단도 여기서 나옴 (Q3와 연결)
  · 회사에서 서비스 전체 만족도는 보통 **매출·사용자 수**로 보거나, **실제 유저 인터뷰**를 함
  · **"주변에 유저 인터뷰 해보시는 것도 방법"**

  □ 구글폼 5문항 + 추첨 인센티브로 직접 피드백 수집 (가장 효율)

▸ 멘토 답변 — **"둘을 섞어라"**

    → PDF 분석(Document Parse) 방향은 유지가 맞음
  · **"이런 걸 쓸 땐 결국 정해진 규칙이 필요하다"** — 예: **이수·과목·학점의 합**
  · **"그런 것들을 모두 LLM에게 맡기기에는 위험하다."**

      [녹취: "NLL" → LLM]
  · **결론 구조**:
      **"문서를 읽는 것은 그대로 하고(DP로 확장), 그 안에서 학점이나 규칙을 뽑아
        계산할 때는 실제 코드로 계산하게끔 구현하면 된다."**

      - **Information Extract API** 또는 **Document Parse API 뒤에 기능을 붙여서** 구조화
        [녹취: "인포메이션 인스펙트", 현장메모 "informance dis" → **Information Extract**]

        → **JSON으로 구조화 정리** (학점: 3, 전공: true)
        → **전공이 true인 것만 나중에 실제로 코드로 더하기**
      - **"바이브 코딩을 할 때는 Information Extract로 필요한 정보를 JSON으로 정리하고,
         실제 계산이 필요한 부분은 코드 기반으로 계산"**
      - 이점: **코드 레벨에서 수정도 쉬워진다**

▸ 액션
  □ 파이프라인 재설계: **DP/Information Extract → JSON 구조화 → 코드 계산**
  □ 학점 합계·전공 이수학점 계산을 Solar에서 **코드로 이관** (P0)
  □ 복수전공은 **규칙 DB 전면 구축보다 PDF 분석 확장 + 코드 계산**으로

【Q7】 AI에게 어디까지 맡겨야 하나 (역할 분담)
───────────────────────────────────────
▸ 우리 질문
  시간 겹침·학점 합계처럼 절대 틀리면 안 되는 건 코드로, "아침 수업 싫어요" 같은
  취향 해석·추천 이유는 AI로 — 이 분담이 맞나?

▸ 멘토 답변 (22:02~23:01) — **분담 방향 맞음. 단 AI 출력을 "구조화된 값"으로 받아라**
  · Q6 답변이 그대로 근거: **계산은 코드, 해석은 AI**가 맞다.
  · "아침 수업 싫어요" 같은 것도 **AI가 직접 시간표를 고르게 두지 말고**,
    **구조화된 형태(딕셔너리)로 표현**하라 [녹취: "미셔너리" → dictionary 추정]
  · **구체적 구성**:
      - 시간표를 **집계된 구조**로 표현 — 예: `{아침: 0, 점심: 2, 저녁: 5}`
      - 사용자가 "아침 수업 싫어요"라고 하면
        → **"아침은 -1을 한다"** 같은 **가중치/점수 규칙으로 변환**
        → 그 규칙을 **코드가 적용해서 조합을 정렬·필터**
  · 즉 **AI = 자연어 요청 → 규칙(숫자)으로 번역** / **코드 = 그 규칙으로 계산·정렬**

▸ 액션
  □ 사용자 취향 → **점수 규칙(JSON)으로 변환**하는 레이어 신설
  □ 시간표를 시간대별 집계 구조로 표현 (아침/점심/저녁 카운트 등)
  □ 조합 정렬·필터는 100% 코드
  □ AI는 규칙 번역 + 추천 이유 설명만
