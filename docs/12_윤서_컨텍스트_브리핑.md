■ 사용자 피드백 담당(윤서)에게 전달하는 컨텍스트 + 시작 프롬프트
════════════════════════════════════════════════════════════════
용도: 이 문서 전체를 복사해서 본인이 쓰는 LLM(ChatGPT/Gemini/Claude 등)에
      붙여넣고 작업을 시작하면 됩니다.
과제: 구글 애널리틱스(GA) 활용 / 지인 직접 피드백 / 구글폼 — 이 중 방식을
      직접 정하고, ① 어떤 과정으로 피드백을 도출할지, ② 그 피드백을
      어떻게 실제 개선으로 반영해나갈지까지 설계하는 것.
최종 갱신: 2026-07-17


────────────────────────────────────────
[A] 프로젝트가 뭔지 (사실)
────────────────────────────────────────
- 서비스: 성균관대 시간표 추천 웹앱. "인텐시브 워크숍(with Upstage)" AI
  Document Builders Challenge 출품작.
- 배포 주소: https://skku-dule.vercel.app/
- 데모데이: **2026-07-25** (오늘 기준 약 8일 남음 — 일정이 촉박함을
  감안해서 방식을 골라야 함)
- 핵심 가치: Upstage의 Document Parse/Extract/Solar로 학생이 올린 문서
  (수강내역 PDF, 졸업요건충족현황 캡처 이미지)를 분석해서 시간표를 짜준다.
- 서비스 흐름(한 페이지에 순서대로): 기본정보 입력 → 학사문서 업로드·분석
  (수강내역/졸업요건) → 과목 선택·시간표 조합 생성 → AI 시간표 추천
  (가중치 설정) → 완성된 시간표 이미지 저장


────────────────────────────────────────
[B] 지금 데이터 수집 현황 (사실)
────────────────────────────────────────
- **구글 애널리틱스(GA)/Microsoft Clarity 등, 어떤 분석 도구도 아직
  연결되어 있지 않습니다.** (코드에서 확인함 — GA 관련 코드 없음)
- **실사용자 테스트를 아직 한 번도 해본 적이 없습니다.** 지금까지는
  팀원들끼리만 써봤습니다. 이건 멘토가 P0(발표 전 필수)로 지적한 항목이기도
  합니다 — "5단계 흐름이 긴지, 어디서 사용자가 피곤한지"조차 아직 판단
  근거가 없는 상태.
- 즉 윤서님의 작업이 **이 서비스가 처음으로 실제 사용자 데이터를
  받아보는 계기**가 됩니다.


────────────────────────────────────────
[C] 지켜야 하는 제약
────────────────────────────────────────
- 이 서비스는 사용자가 업로드하는 문서(수강내역, 졸업요건)를 서버에
  저장하지 않는다는 원칙을 이미 갖고 있습니다. 피드백 수집(설문·인터뷰·GA)도
  이 원칙과 부딪히지 않아야 합니다 — 즉 설문에 이름·학번 같은 민감정보를
  꼭 필요하지 않다면 받지 않는 게 안전합니다.
- GA처럼 **사이트에 코드를 삽입해야 하는 방식**을 고르면, 그 삽입 자체는
  재성이 Claude Code로 처리합니다. 윤서님은 "무엇을 추적할지"(예: 어떤
  버튼 클릭, 어떤 단계 진입/이탈)만 정해서 프롬프트로 넘기면 됩니다 —
  직접 코드를 짤 필요는 없습니다.
- 일정이 촉박합니다(약 8일). 방식을 고를 때 "설치·설계에 걸리는 시간"과
  "결과가 나오기까지 걸리는 시간"을 둘 다 고려해야 합니다.


────────────────────────────────────────
[D] 이번 작업의 배경 — 멘토 피드백 + 팀 회의 조건
────────────────────────────────────────
(2026-07-17 중간점검 멘토링 전체 기록: docs/09_중간피드백_정리.md, Q5)

멘토 피드백 요지:
- "가장 쉬운 건 사실 직접 물어보는 것" — 구글폼 5문항 정도 + (여력 되면)
  추첨 인센티브(기프티콘 등)를 걸면 응답률이 올라간다고 언급.
- GA/Clarity의 실제 쓰임새는 **사용자 행동 퍼널 분석**(어디서 체류하고,
  어디서 이탈하고, 어디서 에러가 나는지). 멘토가 예시로 든 의사결정 흐름:
  > "AI 추천 단계에서 이탈이 많더라(GA로 확인) → 그 단계 체류시간이
  > 5초를 못 넘기더라 → 그래서 서비스팀이 '정확도를 조금 희생하더라도
  > AI 추천 속도를 개선하자'는 식으로 의사결정에 활용한다."
- 서비스 전체 만족도는 보통 매출·사용자 수로 보거나, 실제 유저 인터뷰로
  파악한다 — "주변 지인 인터뷰도 방법"이라고 직접 언급.
- 이 항목은 **P2(여력 되면)**로 분류되어 있음 — P0/P1(실사용 테스트,
  개인정보 고지, Solar 체크리스트 고정 등)이 더 급함. 다만 "피드백 분석
  → 개선"이라는 서사를 발표에 넣으면 설득력이 강해진다고도 언급함.

팀 회의에서 윤서님께 배정된 조건 (원문 그대로):
> "(윤서) 구글 애널리틱스 활용 (시간이 필요함) — 공부해야 할 수도?
> or 지인들 직접 피드백 or 구글폼"

(선택 과제로 같이 언급된 것: 복수전공·연계전공·학과별 교육과정 로드맵
데이터 연동 조사 — 이건 이번 피드백 과제와 별개이니, 여유 있으면
참고만 하세요.)


────────────────────────────────────────
[E] 참고 의견 (Claude Code 제안 — 확정 아님, 하나의 관점일 뿐)
────────────────────────────────────────
⚠️ 아래는 재성님이 작업할 때 쓰는 AI(Claude Code)가 낸 의견입니다.
**정답이 아니라 "이런 방향도 있다"는 참고 자료입니다.** 윤서님이 쓰실
LLM은 아래 내용에 얽매이지 말고, 직접 비교해서 본인만의 결론을
내려주세요. 아래와 다른 결론이어도 전혀 상관없습니다.

간략 요약: 데모데이까지 약 8일밖에 안 남았다는 점을 가장 크게 고려할
만합니다. **구글폼(+가능하면 지인 인터뷰 몇 건 병행)을 1차로, 시간이
남으면 GA를 보조로 얹는** 순서를 고려해볼 만합니다. 이유는 폼/인터뷰는
며칠 안에 "결과"가 나와서 발표 서사("피드백 수집 → 개선")를 만들기
좋은 반면, GA는 **설치 자체는 코드 몇 줄로 빠르지만, 의미 있는 분석을
하려면 실제 사용자 트래픽이 어느 정도 쌓여야** 해서 촉박한 일정에서는
데모 전까지 분석할 데이터가 충분히 안 모일 위험이 있기 때문입니다.

대안 비교 (우열 없음, 그냥 후보 — 실제로는 병행도 가능):
| 방식 | 걸리는 시간 | 얻는 것 | 위험 |
|---|---|---|---|
| 구글폼 5문항 (+추첨 인센티브) | 설계 반나절, 응답 수집 2~3일 | 정성적 만족도·불편 지점, 응답자 본인 언어로 표현된 문제 | 표본이 지인 위주로 편향될 수 있음, "어디서" 이탈했는지는 못 잡음 |
| 지인 직접 인터뷰 몇 건 | 섭외+진행 1~3일 | 폼보다 깊은 "왜"까지 파악 가능 | 표본이 매우 적음(2~5명 수준), 시간이 폼보다 더 듦 |
| GA/Clarity 설치·분석 | 설치는 1일 이내, **유의미한 데이터가 쌓이기까진 트래픽에 따라 며칠~그 이상** | 정량적 행동 데이터(이탈 단계, 체류시간, 클릭 경로) — 폼/인터뷰로는 못 잡는 "실제 행동" | 데모 전까지 트래픽이 부족하면 분석할 데이터 자체가 없을 수 있음 |

**윤서님이 실제로 할 일은 이 표를 그대로 따르는 게 아니라, 이걸 출발점
삼아 본인 판단으로 방식(또는 조합)을 정하고, 왜 그렇게 정했는지 근거를
남기는 것입니다.**

과제: "구글 애널리틱스 활용 / 지인 직접 피드백 / 구글폼" 중 어떤 방식을

조건:
1. E 섹션의 참고 의견·비교표는 하나의 힌트일 뿐이니 반드시 따를 필요
   없다. 동의하지 않으면 왜 다른 방식이 더 낫다고 생각하는지 근거를 대라.
2. 데모데이까지 약 8일이라는 일정 제약을 반드시 고려해라 — "이론적으로
   좋은 방법"이 아니라 "이 기간 안에 실제로 결과를 낼 수 있는 방법"을
   우선해라.
3. 고른 방식에 맞는 구체적인 산출물을 만들어줘:
   - 구글폼을 고른다면: 실제 질문 문항 5개 초안 (객관식/주관식 섞어서,
     인센티브 필요 여부 포함)
   - 인터뷰를 고른다면: 질문 스크립트, 대상 섭외 방법, 진행 시간
   - GA/Clarity를 고른다면: 무엇을 추적할지(어느 버튼, 어느 단계 진입/
     이탈, 어느 이벤트) 구체적인 목록. 코드는 재성이 짤 것이므로, "이런
     이벤트를 추적하고 싶다"는 요구사항 형태로 정리
4. 피드백을 어떻게 "분석"해서 "우선순위를 매길지" 방법을 제시해라
   (예: 응답을 어떤 기준으로 취합하고, 뭘 먼저 고칠지 어떻게 정할지).
5. 분석 결과를 실제 개선으로 이어가는 과정을 설계해줘 — 특히 "이 피드백을
   재성에게 어떤 형태의 프롬프트로 전달할 것인지"까지 포함해라. (재성은
   Claude Code로 구현하는 사람이므로, 모호한 감상이 아니라 "무엇을 어떻게
   바꿔달라"는 구체적 요청으로 정리되어야 함)
6. 결과물 마지막에는, 재성에게 그대로 전달할 수 있는 "실행 계획 요약 +
   첫 번째 실행 프롬프트"를 하나 첨부해줘.
