## 1. 선택한 방식: 구글폼(주력) + GA4·Clarity 동시 설치(보조) + 지인 인터뷰 2~3건(선택)

### 왜 이 조합인가

E 섹션의 참고 의견(구글폼 1차, 시간 남으면 GA)에 대체로 동의하지만, **한 가지는 다르게 판단**했다:

> GA/Clarity를 "시간 남으면 나중에"가 아니라 **오늘(D-7) 바로 설치**해야 한다.

이유:
1. **설치 비용이 거의 0이다.** 코드 삽입은 재성님이 Claude Code로 처리하므로, 윤서 입장에서 드는 비용은 "추적할 이벤트 목록을 넘기는 것"뿐이다. 이건 이 문서에 이미 다 정리했다(3절).
2. **구글폼 배포 자체가 트래픽을 만든다.** 폼을 돌리려면 어차피 서비스 링크를 뿌려야 한다. 그 방문자들이 곧 GA/Clarity 데이터가 된다. 설치를 미루면 이 첫 사용자들의 행동 데이터를 통째로 버리는 셈이다. **먼저 설치 → 그다음 배포** 순서면 같은 노력으로 정성(폼)+정량(GA) 데이터를 동시에 얻는다.
3. **트래픽이 적어도 Clarity는 유의미하다.** E 섹션의 우려("트래픽 부족하면 분석할 데이터가 없다")는 GA의 통계적 퍼널 분석에는 맞는 말이지만, Microsoft Clarity의 **세션 녹화·히트맵은 사용자 10명만 있어도 한 명 한 명의 막히는 지점을 눈으로 볼 수 있다.** 표본이 작을수록 오히려 "통계"보다 "녹화"가 강력하다.
4. **발표 서사가 강해진다.** "설문에서 ○단계가 불편하다는 응답이 많았고(정성), 실제 행동 데이터에서도 그 단계 이탈이 확인됐다(정량), 그래서 이렇게 고쳤다(개선)" — 멘토가 예시로 든 의사결정 흐름을 그대로 재현할 수 있다.

인터뷰는 별도 섭외 없이, **폼에 답해준 지인 중 흥미로운 답변을 한 2~3명에게 10분 통화/대면으로 "왜 그랬는지"만 파고드는** 경량 방식으로 한다. (풀 인터뷰 스크립트·섭외는 D-7 일정에서 시간 대비 효율이 낮음)

## 2. 구글폼 초안 (5문항 + 선택 1문항)

**폼 제목:** 성균관대 시간표 추천 서비스 사용 후기 (1분, 기프티콘 추첨 🎁)

**폼 머리말(안내문):**
> 아래 링크에서 시간표를 한 번 만들어보신 뒤 답해주세요! (약 3~5분)
> 👉 https://skku-dule.vercel.app/
> 응답은 서비스 개선에만 사용되며, 이름·학번은 받지 않습니다.
> 응답자 중 추첨으로 N분께 커피 기프티콘을 드립니다. (당첨 연락용 연락처는 마지막에 '선택'으로만 받으며, 추첨 후 즉시 파기합니다.)

| # | 유형 | 문항 |
|---|------|------|
| 1 | 객관식(단일) | 시간표를 만드는 과정 중 **가장 불편하거나 헷갈렸던 단계**는 어디였나요?<br>① 기본정보 입력 ② 학사문서 업로드·분석 ③ 과목 선택·조합 생성 ④ AI 시간표 추천 ⑤ 이미지 저장 ⑥ 불편한 단계 없었음 ⑦ 중간에 그만둬서 끝까지 못 감 |
| 2 | 척도(1~5) | 업로드한 문서(수강내역/졸업요건)의 **분석 결과가 실제 내 정보와 얼마나 일치**했나요? (1=거의 틀림 ~ 5=완벽히 일치 / 문서를 업로드하지 않았다면 "해당 없음") |
| 3 | 척도(1~5) | AI가 추천해준 시간표를 **다음 학기에 실제로 참고할 의향**이 있나요? (1=전혀 없다 ~ 5=그대로 쓰고 싶다) |
| 4 | 주관식(단답) | 쓰다가 "**여기서 그만두고 싶다**"고 느낀 순간이 있었다면, **언제·왜**였나요? (없었다면 "없음") |
| 5 | 주관식(단답) | 딱 **한 가지만 고칠 수 있다면** 무엇을 고쳐달라고 하시겠어요? |
| 6(선택) | 단답 | (기프티콘 추첨용, 선택) 당첨 연락을 받을 카카오톡 오픈채팅 링크 또는 이메일 — *추첨 후 즉시 파기* |

설계 의도:
- 1번이 핵심 문항 — 5단계 흐름 중 어디가 병목인지(멘토 P0 질문에 직접 답함). ⑦ 선택지로 "완주 실패자"도 잡는다.
- 2번은 Upstage 문서 분석(서비스의 핵심 가치)의 체감 정확도를 따로 측정.
- 4·5번 주관식이 인터뷰 후보 선별 + 개선 프롬프트의 원재료가 된다.
- 이름·학번 등 민감정보 미수집 → [C] 제약 준수. 연락처는 선택 + 파기 고지.

**목표 응답 수: 15~30명** (학과 단톡, 동아리방, 에브리타임 등에 배포)

## 3. GA4 + Clarity 추적 이벤트 목록 (재성님에게 넘길 요구사항)

코드는 재성님이 작성. 아래는 "무엇을 추적하고 싶은지"의 요구사항이다.

### 3-1. 도구
- **GA4**: 이벤트·퍼널 정량 데이터
- **Microsoft Clarity**: 세션 녹화·히트맵 (무료, 스크립트 1줄) — 저트래픽 상황의 핵심 도구

### 3-2. 추적 이벤트

| 이벤트명 | 시점 | 파라미터 | 알고 싶은 것 |
|---|---|---|---|
| `step_enter` | 5단계 각 단계 진입 시 | `step`(1~5), `step_name` | 단계별 도달률 → 퍼널 이탈 지점 |
| `doc_upload_start` | 문서 업로드 시작 | `doc_type`(수강내역/졸업요건) | 업로드 시도율 |
| `doc_parse_success` / `doc_parse_fail` | 문서 분석 완료/실패 | `doc_type`, (실패 시) `error_type` | 분석 성공률, 실패 원인 분포 |
| `ai_recommend_click` | AI 추천 버튼 클릭 | — | 추천 기능 사용률 |
| `ai_recommend_done` | 추천 결과 표시 완료 | `duration_ms`(로딩 소요시간) | **로딩이 몇 초 걸리는지** — 멘토 예시("체류 5초")를 실제로 검증 |
| `weight_adjust` | 가중치 설정 변경 | `weight_type` | 가중치 기능을 실제로 만지는지 |
| `timetable_created` | 완성된 시간표가 화면에 **3초 이상 표시**된 시점 (세션당 1회) | — | **완주율(성공 지표)** — 저장 버튼을 안 눌러도 시간표를 실제로 얻었으면 완주로 인정 |
| `timetable_save` | 시간표 이미지 저장 클릭 | — | 저장률 = 만족·실사용 의향 신호 (완주자 중 몇 %가 저장까지 하나) |
| `field_focus` | 입력 필드에 커서가 들어갈 때 | `field_name`, `step` | 어떤 필드까지 진행했는지 |
| `field_complete` | 필드를 채우고 벗어날 때(blur, 값 있음) | `field_name`, `step` | 필드별 완료율 |
| `form_abandon` | 제출 없이 페이지를 떠날 때 (visibilitychange/beforeunload) | `last_field_focused`, `step` | **정확히 어떤 입력 항목에서 이탈했는지** |

> 💡 필드 단위 이탈은 GA 콘솔 설정으로는 안 잡히고 위처럼 커스텀 이벤트가 필요하다.
> GA4에서 보는 법: **탐색(Explore) → 유입경로(퍼널)** 에 `field_complete`를 필드 순서대로
> 나열하면 어느 필드에서 완료율이 뚝 떨어지는지 보이고, `form_abandon` 이벤트를
> `last_field_focused` 파라미터로 쪼개 보면 이탈 시점의 필드 분포가 바로 나온다.
> (파라미터는 GA4 관리 → 맞춤 정의에 '맞춤 측정기준'으로 등록해야 보고서에 뜸)
> 트래픽이 적은 초반엔 Clarity 세션 녹화로 이탈 직전 화면을 직접 보는 게 더 빠르다.

### 3-3. 보고 싶은 지표 (수집 후)
1. 퍼널: 방문 → step 1 → 2 → 3 → 4 → 5 → `timetable_created`(완주) → `timetable_save`(저장) 각 구간 전환율
2. `doc_parse_fail` 비율과 실패 유형
3. `ai_recommend_done`의 `duration_ms` 분포 (추천 대기시간)
4. Clarity: 이탈 세션 녹화 — 특히 문서 업로드 단계와 AI 추천 대기 중 행동

## 4. 분석 → 우선순위 방법

**수집 마감: D-4 (7/21 밤)** 에 1차 마감하고 분석 시작.

1. **주관식 태깅**: 4·5번 응답을 전부 `[단계번호] + [유형: 오류/느림/헷갈림/기능부재] + [심각도]`로 태깅한다. 심각도는 3단계:
   - **Blocker** — 이것 때문에 완주를 못 함 (1번 문항 ⑦ 응답자, "그만두고 싶었다" 응답)
   - **Friction** — 완주는 했지만 짜증/혼란
   - **Nice-to-have** — 있으면 좋겠다 수준
2. **정량 교차검증**: 태깅 결과 상위 단계를 GA 퍼널 이탈률·Clarity 녹화와 대조한다. "설문에서도, 행동 데이터에서도 같은 단계가 문제"면 확신을 갖고 우선순위 최상단에 놓는다.
3. **우선순위 공식**: `빈도 × 심각도`로 정렬한 뒤, 재성님에게 각 항목의 **수정 난이도(상/중/하)**만 물어봐서 최종 순서를 정한다.
   - 1순위: Blocker + 다빈도 + 난이도 하~중 (데모 전 반드시 수정)
   - 2순위: Friction + 다빈도 (되면 수정)
   - 3순위: 나머지 → 발표 자료의 "향후 개선 계획" 슬라이드로 보냄 (버리는 게 아니라 서사 재료로 씀)

## 5. 피드백 → 개선 파이프라인 (재성님에게 전달하는 형식)

개선 요청은 반드시 아래 템플릿으로 전달한다. 모호한 감상("업로드가 좀 별로래요") 금지.

```
[개선 요청 #N] — 우선순위: 1순위/2순위

■ 현상 (데이터):
  설문 응답 15명 중 6명이 "문서 업로드·분석" 단계를 가장 불편하다고 답함.
  GA에서 doc_parse_fail 비율 30%, Clarity 녹화에서 실패 후 재시도 없이 이탈 3건 확인.

■ 원인 추정:
  분석 실패 시 에러 메시지가 "실패했습니다"뿐이라 사용자가 뭘 잘못했는지 모름.

■ 요청 사항 (구체적으로):
  1. 분석 실패 시 실패 원인별 안내 문구 표시 (예: "PDF가 아닌 파일입니다",

  2. 실패 화면에 "다시 업로드" 버튼 추가

■ 완료 기준:
  잘못된 파일을 올렸을 때 원인 안내 + 재시도 버튼이 보이면 완료.
```

## 7. GA로 볼 개선 지표 카탈로그 — "무엇을 재면 어떤 개선 결정이 나오는가"

각 지표는 `측정 방법 → 결과가 이렇게 나오면 → 이렇게 개선한다` 구조로 정리.
(★ = 3절의 커스텀 이벤트 필요 / 무표시 = GA4 기본 제공)

### A. 흐름(퍼널) 지표 — "어디서 떠나는가"

| 지표 | 측정 방법 | 개선 판단 예시 |
|---|---|---|
| ★ 단계별 이탈률 | `step_enter`(1~5) → `timetable_save` 퍼널 | 2→3단계 이탈 40% 이상이면 문서 업로드 단계가 병목 → 업로드 UX 최우선 수정 |
| ★ 입력 항목별 이탈률 | `form_abandon`의 `last_field_focused` 분포 | 특정 필드(예: 학번/학과 선택)에 이탈이 몰리면 → 그 필드를 선택형으로 바꾸거나 뒤 단계로 미루기 |
| ★ 입력 항목별 완료율 낙차 | `field_complete`를 필드 순서대로 퍼널 배열 | 완료율이 뚝 떨어지는 필드 = 어렵거나 귀찮은 항목 → 필수→선택 전환, 기본값 제공 |
| 단계별 체류시간 | 단계별 평균 참여 시간 (또는 ★`step_enter` 간 시간차) | 체류가 비정상적으로 긴 단계 = 헷갈리거나 느린 단계 → 안내 문구·로딩 표시 추가. 반대로 AI 추천 결과에서 체류 5초 미만 = 결과를 안 읽고 떠남 → 결과 표시 방식 개선 (멘토 예시 그대로) |

### B. 핵심 기능 품질 지표 — "Upstage 분석과 AI 추천이 제값을 하는가"

| 지표 | 측정 방법 | 개선 판단 예시 |
|---|---|---|
| ★ 문서 분석 성공률 | `doc_parse_success` / (`success`+`fail`), `doc_type`별 | 캡처 이미지(졸업요건) 실패율만 높으면 → 이미지 업로드 가이드(예시 스크린샷) 추가 |
| ★ 분석 실패 후 재시도율 | `doc_parse_fail` 후 같은 세션의 `doc_upload_start` 재발생 비율 | 재시도 없이 이탈이 많으면 → 실패 메시지에 원인 안내 + 재업로드 버튼 (개선 요청 #예시와 연결) |
| ★ AI 추천 대기시간 | `ai_recommend_done`의 `duration_ms` 분포 | p50이 5초 넘으면 → 정확도 일부 희생하더라도 속도 개선 or 스켈레톤/진행률 UI |
| ★ 추천 재요청 횟수 | 세션당 `ai_recommend_click` 횟수 | 3회 이상 재요청이 흔하면 = 첫 추천이 마음에 안 듦 → 추천 로직·가중치 기본값 점검 |
| ★ 가중치 기능 사용률 | `weight_adjust` 발생 세션 비율 | 10% 미만이면 → 기능이 안 보이거나 어려운 것 → 프리셋(아침수업 없음/공강 몰기 등) 제공 |

### C. 전체 성과·환경 지표 — "누가 어떻게 와서 완주하는가"

| 지표 | 측정 방법 | 개선 판단 예시 |
|---|---|---|
| ★ 완주율(북극성 지표) | `timetable_created` 세션 / 전체 세션 | 발표에서 "개선 전 X% → 개선 후 Y%"로 쓸 핵심 숫자. 저장 클릭이 아니라 "시간표가 화면에 만들어짐" 기준이라 저장 안 누른 완주자도 포함. **설치 직후부터** 쌓아야 함 |
| ★ 저장률(만족 신호) | `timetable_save` / `timetable_created` | 완주자 중 이미지 저장까지 한 비율 — "결과물이 가져갈 만했는가"의 행동 지표. 낮으면 추천 품질·결과 화면 개선 신호 |
| 기기별 완주율 | 기기 카테고리(모바일/PC) × 위 완주율 | 단톡으로 배포하면 대부분 모바일 유입 — 모바일 완주율만 낮으면 반응형 레이아웃부터 수정 |
| 유입 채널별 품질 | 세션 소스/매체 (배포 링크에 `?utm_source=everytime` 등 UTM 붙이기) | 에브리타임 유입은 이탈이 빠르고 학과 단톡 유입은 완주가 많다면 → 남은 기간 배포 채널 선택에 반영 |
| 재방문율 | 신규 vs 재방문 사용자 | 재방문이 있다면 = 실사용 의향 신호 → 발표 서사에 활용 |
| 스크롤 도달률 | 스크롤 90% 도달(기본 제공) | 한 페이지 구조라 아래 단계가 화면 밖에 있음 — 스크롤 도달이 낮으면 단계 진행 시 자동 스크롤 추가 |

### 우선순위 (7일 안에 다 볼 수는 없으므로)

1. **A의 단계별 이탈률 + B의 분석 성공률·추천 대기시간** — 멘토 P0 질문("흐름이 긴지, 어디서 피곤한지")에 직접 답하는 지표. 이번 주 필수.
2. **C의 완주율** — 개선 전/후 비교용으로 첫날부터 쌓기만 하면 됨.
3. A의 필드 단위 지표는 1에서 "1단계(기본정보)가 병목"으로 나왔을 때 파고드는 2차 지표.
4. 트래픽이 적어 숫자가 불안정하면 → 같은 질문을 Clarity 세션 녹화로 확인 (10명이어도 눈으로 보임).
