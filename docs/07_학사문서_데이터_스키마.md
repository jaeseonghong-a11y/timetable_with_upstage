# GLS 학사문서 데이터 정리 방식

최종 갱신: 2026-07-14

대상 문서는 서로 목적이 다르므로 한 테이블로 합치지 않는다.

- **수강/취득과목 리스트**: 무엇을 이미 들었는지 판단하는 `completedCourses`
- **졸업요건충족현황**: 어떤 영역이 얼마나 남았는지 판단하는 `requirements`
- 두 결과는 개인정보를 제외한 `AcademicProfile`에서 합친다.

실제 저장·Solar 출력 계약은 `docs/schemas/academic-profile.schema.json`을 사용한다.

## 1. 공통 원칙

- 이름·전체 학번·연락처·생년월일은 저장하지 않는다.
- 학번 대신 졸업요건 분기에 필요한 **입학연도 4자리**만 사용자가 직접 입력한다.
- 원본 PDF·이미지와 전체 Parse Markdown은 현재 세션에서만 사용하고 영구 저장·로그·커밋하지 않는다.
- Document Parse 결과는 `draft`다. 스키마 검증과 사용자 확인을 통과해야 `confirmed`가 된다.
- Solar는 문서의 불규칙한 표를 고정 JSON으로 정리하지만 졸업 가능 여부를 단독 판정하지 않는다.
- 실제 추천은 `confirmed` 데이터만 사용한다.

## 2. 수강/취득과목 리스트 → `completedCourses`

실제 PDF에서 확인한 열은 다음과 같다.

| 원본 열 | 저장 필드 | 처리 |
|---|---|---|
| 전공 | `majorScope` | 제1전공·제2전공 등 원문 분류 |
| 이수구분 | `classification` | 교양·전공·선택·DS 등 |
| 년도 | `year` | 4자리 정수, 불명확하면 `null` |
| 학기 | `term` | `spring/summer/fall/winter`, 불명확하면 `null` |
| 학수번호+과목명 | `courseCode`, `courseName` | 정규식으로 코드와 이름 분리 |
| 영역 | `area` | 전공코어·전공심화·글로벌 등 |
| 학점 | `credits` | 숫자 |
| 성적 | `completionStatus` | 원점수·등급은 버리고 이수/실패/철회 여부만 저장 |
| 비고·국제어 | `flags` | 추천에 필요한 표지만 제한적으로 저장 |

추가 필드:

- `recommendationPolicy`: 기본 `exclude`, 사용자가 재수강으로 선택하면 `retake`
- `reviewReasons`: 셀 병합·값 누락 등 사용자 확인이 필요한 이유
- `sourceDocumentId`: 어떤 업로드에서 나온 행인지 연결하는 임의 세션 ID

### 추천에서 사용하는 방식

1. `completionStatus=earned`이고 `recommendationPolicy=exclude`인 학수번호는 후보에서 제외한다.
2. 사용자가 재수강 토글을 켜면 기록은 보존하고 `recommendationPolicy=retake`로 바꾼다.
3. 분반 정보는 과거 이수목록에 없으므로 학수번호까지만 비교한다.
4. 0학점 안전교육 같은 과목은 이력에는 남기되 일반 시간표 추천 후보로 사용하지 않는다.
5. 정확한 성적등급은 추천에 사용하지 않으므로 구조화 직후 폐기한다.

## 3. 졸업요건충족현황 → `requirements`

캡처에서 확인한 열은 다음과 같다.

- 구분
- 기준학점
- 취득학점
- 수강학점: 1학기·여름·2학기·겨울·계
- 잔여학점

각 행은 다음 구조로 저장한다.

- `requirementId`: 범위와 행 이름을 합친 내부 식별자
- `scope`: `primary_major/general/ds/university/other`
- `label`: 화면에 표시할 원본 요건명
- `rule`: 숫자 최소학점, 영역분배, 이수여부, 수동확인 중 하나
- `earnedCredits`: 취득학점 숫자, 복합값이면 `null`
- `inProgressCredits`: 학기별 수강학점과 합계
- `remainingCredits`: 잔여학점 숫자
- `status`: `satisfied/in_progress/unmet/review`
- `rawValues`: `6 / 0` 같은 복합 셀 원문
- `reviewReasons`: 자동 해석하지 못한 이유

### 요건 규칙 종류

1. `credit_minimum`: 전공심화 36학점처럼 최소 학점이 숫자인 행
2. `distribution_minimum`: “3개 영역 중 2개 이상, 합계 6학점”처럼 여러 행이 공유하는 영역분배 요건
   - `groupId`: 같은 공동 요건에 속한 행을 묶는 식별자
   - `totalAreas`: 선택 가능한 전체 영역 수
   - `minimumAreas`: 학점을 취득해야 하는 최소 서로 다른 영역 수
   - `totalCredits`: 각 영역별 학점이 아니라 그룹 전체의 합계 최소학점
3. `completion`: 졸업논문·필수인증처럼 학점보다 완료 여부가 중요한 행
4. `manual`: 형식이 불명확해 사용자 확인이 필요한 행

`잔여학점`은 그대로 믿지 않고 숫자형 행에서는 `기준-취득-수강중`으로 재계산한 값과 비교한다.
둘이 다르면 `status=review`로 둔다. `6 / 0`처럼 의미가 확정되지 않은 복합값도 추측하지 않는다.

균형교양은 `groupId=balanced-general`, `totalAreas=3`, `minimumAreas=2`, `totalCredits=6`으로 저장한다.
따라서 영역별 학점이 `[6, 0, 0]`이면 불충족이고, `[3, 3, 0]`·`[1, 5, 0]`·`[0, 3, 3]`은
충족이다. 두 영역을 들었더라도 `[2, 3, 0]`처럼 합계가 6학점 미만이면 불충족이다. 즉, 세 영역
각각에서 6학점을 요구하거나 특정 두 영역을 고정하는 규칙이 아니다.

캡처 안내처럼 C/L 과목은 여러 전공의 취득학점에 중복 표시될 수 있다. 따라서 `requirements` 행끼리
취득학점을 더해 전체 취득학점을 만들지 않는다. 요건 진행률은 범위별 행을 독립적으로 보존하고,
기수강 과목 목록에서만 정규화된 학수번호 기준으로 중복을 제거한다.

## 4. 실제 Parse에서 확인된 오류와 대응

실제 수강/취득과목 PDF는 Document Parse HTTP 200, Markdown 약 3.7KB, 요소 8개로 변환됐다.
표 헤더와 대부분의 과목 행은 읽혔지만 다음 문제가 있었다.

- 연도와 학기가 한 칸에 붙거나 순서가 바뀜
- 한 행에 학수번호·과목명이 두 개 합쳐짐
- `제1전공` 같은 셀 값이 중복됨
- 영역·학점·성적이 옆 열로 이동함
- 중간 합계 행이 일반 과목 행처럼 보임

따라서 다음 검증을 적용한다.

- 학수번호는 실제 GLS 표기처럼 `[A-Z]{2,6}[0-9]{3,4}` 형식이어야 한다.
- `1학기/2학기/여름학/겨울학`과 열 경계에서 갈라진 `2 | 학기` 표기를 네 개의 내부 학기로 정규화한다.
- 한 셀에 서로 다른 `학수번호+과목명` 쌍이 2개 이상 있으면 각각의 draft로 분리한다. 코드와 이름이
  명확히 대응하면 분리 자체는 사용자 확인 사유로 만들지 않는다.
- `취득학점:`·`합계:`가 있는 행은 과목으로 만들지 않는다.
- 연도는 4자리, 학기는 허용된 네 값, 학점은 0 이상 숫자인지 검증한다.
- Document Parse 표에서 코드·과목명이 명확한 행은 Solar가 누락하거나 부가 열을 잘못 반환해도 과목
  자체를 버리지 않는다. 표의 코드·이름·학기를 기준으로 보완하고 Solar 값은 전공범위·이수구분 등
  부가정보를 보강하는 데 사용한다.
- 표 코드 대비 Solar 결과가 부족하면 누락 코드만 명시해 한 번 재추출하고, 재시도가 실패해도 첫 결과와
  결정론적으로 보완한 표 결과를 유지한다.

2026-07-14 원본 PDF 실측에서 Document Parse 표는 32개 행을 만들었지만 한 행에 서로 다른 두 과목이
합쳐져 실제 학수번호·과목명 쌍은 33개였다. 최종 서비스 API 재검증 결과는 HTTP 200, 기수강 과목 33개,
정규화된 학기 33개, 잘못된 행 0개, 전역/과목별 검토 사유 0개였다. 원본·전체 Parse/Solar 결과는
저장하거나 커밋하지 않았다.

## 5. 사용자 확인 화면

문서 분석 직후 두 개의 편집 가능한 표를 보여 준다.

### 기수강 과목

- 체크됨: 추천에서 제외
- `재수강` 토글: 다시 추천 후보로 허용
- `수정`: 학수번호·과목명·학점·영역 수정
- `삭제`: 잘못 추출된 행만 세션 데이터에서 제거

### 졸업요건

- 충족·수강중·미충족·확인 필요 상태
- 기준/취득/수강중/잔여 수치
- 영역분배 규칙의 원문과 해석 결과
- 확인 필요 행은 사용자가 승인하기 전 추천에 사용하지 않음

## 6. 추천 엔진에 전달할 최소 데이터

Solar와 조합 엔진에는 다음만 전달한다.

- 소속 학과·전공·트랙 코드, 입학연도, 현재 학년, 주 캠퍼스
- 확정된 기수강 학수번호와 재수강 대상 학수번호
- 확정된 미충족 요건과 남은 학점/영역
- 이번 학기 실제 개설강좌와 교과과정 후보
- 사용자 필수과목·책가방·고정일정·성향

이름, 전체 학번, 정확한 과거 성적, 원본 문서는 전달하지 않는다.
