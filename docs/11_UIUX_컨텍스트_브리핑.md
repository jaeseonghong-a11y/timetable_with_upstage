[B] 지금 화면 흐름 (실제 코드 기준)
────────────────────────────────────────
지금은 아래 내용이 **한 페이지에 위에서 아래로 쭉 이어지는 구조**입니다.
"다음" 버튼으로 화면이 통째로 넘어가는 방식이 아니라, 스크롤을 내리면서
순서대로 나타납니다.

1. **기본 정보 입력** — "어떤 강좌를 찾을지 먼저 알려 주세요"
   (학과·학년·학기 등 검색 조건)
2. **학사문서 업로드 및 분석** — "학사문서를 읽고, 반드시 본인이 확인합니다"
   - 수강내역(취득과목) PDF 업로드 → AI 분석 → 사용자가 결과 확인/수정 → 확정
   - 졸업요건충족현황 캡처 이미지 업로드 → AI 분석 → 확인 → 확정
   (이 두 문서는 서로 독립적인 카드로, 순서 상관없이 아무 때나 올릴 수 있음)
3. **시간표 계획**
   - 필수 과목 / 선택 과목(택 N) 설정
   - 유효 시간표 조합 자동 생성 (여러 개, 순위 없음 — 겹치는 시간 없이
     가능한 조합을 전부 보여줌)
   - AI 시간표 추천 (사용자가 가중치를 설정하면, Solar가 상위 후보를 골라
     추천 이유를 붙여줌)
   - 완성된 시간표를 이미지로 저장하는 기능

[C] 지켜야 하는 제약 (벗어나면 안 됨)
────────────────────────────────────────
- 실제 로직(API 호출, 상태 관리, 데이터 처리)은 건드리지 않습니다. UI/UX
  작업은 "보이는 것 / 전환되는 방식"만 다룹니다.
- 스타일은 컴포넌트별 `*.module.css` 파일에만 있습니다. 실제 구현은
  재성이 하지만, 디자인 제안도 이 구조(컴포넌트 단위)를 기준으로 하면
  나중에 옮기기 쉽습니다.
- Upstage 문서 분석에는 실제로 몇 초~십수 초가 걸립니다(특히 문서 분석
  단계). 로딩 자체를 없앨 수는 없고, **"덜 지루하게 느껴지게"** 하는 게
  목표입니다.
- API 키, 서버 로직, 데이터 저장 방식은 UI/UX 범위 밖입니다 (참고로,
  사용자 문서는 서버에 저장되지 않고 분석 후 즉시 처리됩니다).


────────────────────────────────────────
[D] 이번 작업의 배경 — 멘토 피드백 + 팀 회의 조건
────────────────────────────────────────
(2026-07-17 중간점검 멘토링 전체 기록: docs/09_중간피드백_정리.md)

멘토 피드백 중 UI/UX 관련 항목:
- "5단계 흐름이 긴지, 사용자가 어디서 피곤해하는지"는 **아직 실사용
  테스트를 안 해봐서 모른다** — 판단 근거 자체가 없다는 게 멘토의 지적.
  (→ 규동님 제안도 "느낌"이 아니라 실제 사용자 흐름을 가정한 근거가
  있으면 좋습니다.)
- 로딩 중 사용자가 지루하지 않게: 진행 표시, 스트리밍 노출, 혹은 "로딩을
  사용자 작업 뒤로 숨기는" 순서 재배치.
- 참고 사례로 언급된 것: ChatGPT의 "생각 중…" 스트리밍 — 실제 답변이
  10초 걸려도 스트리밍으로 보여주면 체감 대기시간이 훨씬 짧다는 지적.
- (참고, UI/UX 범위는 아니지만 화면에 자연스럽게 얹어야 할 수 있는 것)
  재미난 기능 후보: 특정 교수 제외, 친구와 같이 듣기, 공강 확보, 특정
  요일 수업 제외.

팀 회의에서 규동님께 배정된 조건 (원문 그대로):
> "UI/UX 개선 — 사례조사·레퍼런스 등 (추가될 기능까지 고려)(로딩 시간까지
> 고려)(단계별 구분이 확실하게 — 간격을 주거나, 서서히 넘기거나, 창이
> 넘어가게)"


────────────────────────────────────────
[E] 이번에 같이 고려할 것: 구글 애널리틱스(GA)
────────────────────────────────────────
GA는 아직 연결되어 있지 않습니다(다른 조원 담당, 별도 작업). 하지만 붙일
예정임을 감안하면:
- 화면 흐름이 "명확하게 구분되는 단계"일수록, 나중에 "몇 번째 단계에서
  사용자가 이탈하는지"를 GA로 분석하기 쉬워집니다.
- 반대로 지금처럼 한 페이지가 쭉 이어져 있으면, 어디서 이탈했는지
  구분하기 어렵습니다.
- 즉 이번 UI/UX 개선 방향과 향후 GA 분석 방향이 서로 맞물려 있습니다.
  디자인할 때 "이 화면이 GA에서 하나의 이벤트/구간으로 잡힐 수 있는가"를
  같이 염두에 두면 좋습니다.

[F] 참고 의견 (Claude Code 제안 — 확정 아님, 하나의 관점일 뿐)
────────────────────────────────────────
⚠️ 아래는 재성님이 작업할 때 쓰는 AI(Claude Code)가 낸 의견입니다.
**정답이 아니라 "이런 방향도 있다"는 참고 자료입니다.** 규동님이 쓰실
LLM은 아래 내용에 얽매이지 말고, 직접 리서치해서 스스로 비교하고 본인만의
결론을 내려주세요. 아래와 다른 결론이어도 전혀 상관없습니다 — 오히려
독립적인 관점이 더 도움이 됩니다.

간략 요약: 지금의 "한 페이지 이어짐" 구조를 **단계별로 화면이 전환되는
위저드(wizard) 방식**으로 바꾸는 방향을 고려해볼 만합니다. 이유는 (1)
로딩을 다음 단계로 넘어가는 전환 애니메이션 뒤에 자연스럽게 숨길 수 있고,
(2) 나중에 GA를 붙일 때 단계별 이탈률을 측정하기 쉬워지기 때문입니다.
다만 이건 지금 구조 대비 엔지니어링 변경 범위가 있어서 재성님과 상의가
필요합니다.

같이 생각해볼 다른 대안들 (우열 없음, 그냥 후보):
| 대안 | 방식 | 장점 | 단점 |
|---|---|---|---|
| 스텝 위저드 | 한 화면에 한 단계만, 완료 시 다음 화면으로 전환 | 로딩을 전환 뒤에 숨기기 좋음, GA 측정 쉬움 | 구조 변경 범위 큼 |
| 아코디언 | 한 페이지 유지, 완료된 섹션은 접히고 다음 섹션이 펼쳐짐 | 변경 범위 작음, 이전 단계로 돌아가기 쉬움 | "화면이 넘어간다"는 느낌은 약함 |
| 진행률 바 + 스크롤 앵커 | 구조는 그대로, 상단에 "1/5" 진행 표시만 추가 | 가장 적은 변경, 리스크 낮음 | 로딩을 숨기는 효과는 적음 |
| 슬라이드 전환 | 한 화면에 한 섹션만, 옆으로 슬라이드 | 모바일 앱 느낌, 전환 애니메이션에 로딩 자연스럽게 숨기기 좋음 | 데스크톱에서는 어색할 수 있음 |

**규동님이 실제로 할 일은 이 표를 그대로 쓰는 게 아니라, 이걸 하나의
출발점 삼아 직접 사례조사를 통해 검증하거나 반박하거나 새로운 대안을
추가하는 것입니다.**

너는 UI/UX 디자이너다. 아래는 대학생 대상 시간표 추천 웹서비스의 실제

과제: 이 서비스의 "단계별 흐름"을 어떻게 시각적으로 표현하면 좋을지
설계안을 제안해줘.

조건:
1. 최소 2~3개의 서로 다른 UX 패턴 대안을 각각 제시하고 장단점을
   비교해줘. F 섹션의 표를 참고하되, 거기 없는 다른 패턴도 자유롭게

   동의하지 않으면 왜 다른 대안이 더 낫다고 생각하는지 근거를 대라.
3. 로딩 시간(수 초~십수 초)을 사용자가 덜 지루하게 느끼도록 하는
   요소를 각 대안에 포함해라.
4. 나중에 구글 애널리틱스로 "어느 단계에서 이탈하는지" 측정하기 쉬운
   구조인지도 각 대안별로 평가해라.
5. 최종적으로는 실제 개발자(재성)가 Claude Code라는 AI 코딩 도구로

   상태값을 구체적으로 정리해줘. (직접 코드를 작성할 필요는 없다 —
   설계와 근거 중심으로.)
6. 결과물 마지막에는, 재성에게 그대로 전달해서 Claude Code에 입력할 수
