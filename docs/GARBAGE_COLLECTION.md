# GARBAGE_COLLECTION.md — 주기적 청소 에이전트 절차

> **왜 필요한가:** AI는 기존 코드를 보고 따라 하므로, 나쁜 패턴·죽은 코드·문서-코드 불일치가
> **복리로 불어난다(눈덩이).** 사람이 안 치우면 코드베이스가 썩는다.
> 이 절차는 주기적으로 그 눈덩이를 녹인다.

---

## 언제 실행하나
- 큰 기능(수집기 완성, 파싱 파이프라인 완성 등) 하나가 끝날 때마다
- 또는 주 1회 정기적으로
- 또는 "요즘 코드가 지저분해졌다"는 느낌이 들 때

## 어떻게 실행하나
AI 도구(Claude Code 또는 Codex)에게:
> "GARBAGE_COLLECTION.md 절차대로 청소해줘. 발견한 문제는 먼저 목록으로 보고하고, 내 승인 후 고쳐."

**중요:** 청소 에이전트는 **먼저 보고, 승인 후 수정**한다. 멋대로 대량 삭제하지 않는다.

---

## 점검 체크리스트 (4대 항목)

### ① 문서 ↔ 코드 불일치
- [ ] `AGENTS.md`에 적힌 명령어·스크립트가 실제 `package.json`/`pyproject.toml`과 일치하는가
- [ ] `PROJECT_STRUCTURE.md`의 폴더 구조가 실제와 같은가 (새 폴더/삭제된 폴더 반영됐나)
- [ ] `docs/02_기술검증_기록.md`의 API 명세가 실제 수집기 코드와 일치하는가
- [ ] `docs/00_프로젝트_현황_요약.md`가 실제 진행상황을 반영하는가 (오래된 "예정" 항목 없나)

### ② 규칙 위반 코드
- [ ] API 키 하드코딩이 새로 생기지 않았나 (`grep -ri "api.key\|sk-\|UPSTAGE.*="` 로 스캔)
- [ ] 클라이언트 컴포넌트에서 Upstage 직접 호출하는 코드 없나 (키 노출)
- [ ] `// eslint-disable` / `# noqa` / `# type: ignore` 가 이유 주석 없이 남발됐나
- [ ] 크롤링이 서비스 심장으로 비대해지지 않았나 (Upstage 필연성 유지 확인)
- [ ] 요청 간격 없이 성대 API를 때리는 코드 없나 (서버 부담·법적 안전선)

### ③ 죽은 코드 / 미사용 자산
- [ ] import했지만 안 쓰는 모듈 (린터가 대부분 잡지만 재확인)
- [ ] 정의했지만 아무데서도 호출 안 하는 함수/컴포넌트
- [ ] 주석 처리된 채 방치된 코드 블록 (삭제하거나 살리거나)
- [ ] `package.json`/`pyproject.toml`에 설치했지만 안 쓰는 의존성
- [ ] `scraper/data/`에 오래된 테스트 JSON이 커밋돼 쌓이지 않았나

### ④ 중복 / 복사된 로직
- [ ] 같은 로직이 여러 파일에 복붙됐나 (SSV 파싱, Upstage 호출 등) → 공통 함수로 추출
- [ ] 비슷한 컴포넌트가 미묘하게 다르게 여러 개 생겼나

---

## 청소 후
1. 발견·수정 내역을 `.github/agent-logs/YYYY-MM-DD-gc.md`에 기록
2. 규칙 공백이 드러났으면 `AGENTS.md` 4장에 규칙 추가 + `docs/01_의사결정_로그.md`에 로그
3. 품질 게이트(lint/typecheck/test) 통과 확인 후 커밋: `chore: 가비지 컬렉션 - <요약>`

## 자동화 힌트 (나중에)
- 죽은 코드: `ts-prune`(TS), `ruff`(Python 미사용 import) 로 일부 자동 검출
- 키 스캔: pre-commit에 `gitleaks` 훅 추가 검토
- 문서-코드 일치: CI에서 "AGENTS.md 명령어가 실제 존재하나" 스모크 체크
- 이 자동화들은 코드가 쌓인 뒤(Tier 2) 도입한다. 지금은 수동 체크리스트로 충분.
